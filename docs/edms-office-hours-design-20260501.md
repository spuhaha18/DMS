# Design: QA 설정 중심 EDMS Core

## Problem Statement

제약회사 연구소의 분석법/밸리데이션 문서를 전자적으로 등록, 결재, 서명, 배포, 열람 관리하는 EDMS를 구축한다.

현재 문서는 하드카피 중심으로 관리되고 있으며, QA는 승인 완료본을 스캔한 PDF를 파일서버에서 받아 Acrobat으로 true copy를 만들고, 엑셀 대장으로 문서번호와 승인 여부를 관리한다. 가장 큰 문제는 누가 어떤 공식본을 언제 열람했는지 추적되지 않는다는 점이다.

새 시스템은 하드카피 true copy 자동화가 아니라, 전자 원본 기반 official PDF 생성과 배포 통제를 목표로 한다.

## Demand Evidence

- QA가 현재 Acrobat, 파일서버, 엑셀 대장을 조합해 공식 문서 관리를 수행한다.
- 연구원이 문서를 등록하려 할 때 문서번호가 필요하다.
- 기존에는 문서번호 발급 후 승인되지 않으면 번호를 삭제하는 경우가 있다.
- 문서 종류별 승인자가 달라 수동 라우팅 부담이 있다.
- 현재 열람 이력이 남지 않아 감사/실사 대응 리스크가 있다.

## Status Quo

현재 흐름:

1. 연구원이 문서 작성 전 문서번호를 요청한다.
2. 문서번호는 과제코드 + 문서종류 + 일련번호 체계로 관리된다.
3. 문서는 하드카피로 승인되고 수기로 서명된다.
4. 연구원이 승인 완료본을 스캔해 PDF로 파일서버에 업로드한다.
5. QA가 Acrobat으로 워터마크와 전자도장이 포함된 true copy를 생성한다.
6. QA가 엑셀 대장에 문서번호와 승인 여부를 관리한다.
7. 파일서버에서 열람되지만 열람자는 추적되지 않는다.

목표 흐름:

1. 연구원이 문서 등록을 시작한다.
2. 과제코드와 문서종류를 선택한다.
3. 시스템이 문서번호를 자동 발급한다.
4. 연구원이 doc/docx 파일을 업로드한다.
5. 시스템이 문서 종류별 전자결재 라우트를 적용한다.
6. 작성자, 검토자, 승인자가 전자서명한다.
7. 승인 완료 시 시스템이 official PDF를 자동 생성한다.
8. official PDF에 워터마크를 자동 적용한다.
9. 시스템 안에서 열람/배포하고 모든 열람 이력을 남긴다.

## Target User & Narrowest Wedge

1차 사용자는 연구원이 아니라 QA/문서관리자다.

첫 문서군은 분석법/밸리데이션 doc/docx 문서로 제한한다. 첫 번째 가치는 문서 작성 편의가 아니라 다음 네 가지다.

- 문서번호 자동 발급과 상태 추적
- 문서 종류별 전자결재/전자서명
- 승인 완료본 official PDF 자동 생성
- 권한 기반 열람/배포와 열람 이력 감사추적

## Constraints

- 1차 PDF 변환 대상은 doc/docx만 포함한다.
- xls/xlsx는 1차 official PDF 자동 변환 대상에서 제외한다.
- true copy는 하드카피 기반일 때 필요한 개념이므로 1차 범위에서 제외한다.
- 전자도장은 1차 범위에서 제외한다. 워터마크와 전자서명 이력/manifest를 사용한다.
- Word 웹 편집기, 공동작성, 모바일 결재, 외부 협력사 서명은 제외한다.
- 결재/전자서명은 1차 범위에 포함한다.
- QA는 개별 문서의 official PDF를 수동 확인하지 않는다. QA는 번호 규칙, 문서종류, 결재 라우트, 권한, 예외를 관리한다.

## Premises

1. 문서번호는 연구원이 등록을 시작할 때 시스템이 자동 발급한다.
2. 문서번호 체계는 과제코드 + 문서종류 + 일련번호다.
3. 발급된 문서번호는 취소/반려되어도 재사용하지 않는다.
4. 버전/개정은 문서번호와 분리해 시스템 메타데이터로 관리한다.
5. 승인 완료 후 시스템이 생성한 official PDF가 공식 열람/배포본이다.
6. official PDF 생성에는 전자도장이 필요하지 않다.
7. 열람은 파일서버가 아니라 시스템 안에서 수행되고 이력이 남아야 한다.

## Approaches Considered

### Approach A: 최소 전자결재 DMS

doc/docx 업로드, 자동 문서번호, 전자결재/전자서명, official PDF 생성, 열람 이력만 구현한다. QA 설정 기능은 최소화하고 분석법/밸리데이션 문서 종류를 거의 고정한다.

장점:

- 가장 빨리 내부 데모와 PoC를 만들 수 있다.
- 번호 발급, 승인 이력, 공식 PDF, 열람 추적 문제를 직접 해결한다.
- CSV 범위를 상대적으로 작게 유지할 수 있다.

단점:

- 문서 종류나 결재 라우트가 늘어나면 빠르게 한계가 온다.
- QA가 운영 중 설정을 바꾸기 어렵다.
- 나중에 확장할 때 구조 변경 가능성이 높다.

### Approach B: QA 설정 중심 EDMS Core

QA가 문서종류, 번호 규칙, 결재 라우트, 권한, 워터마크 규칙을 관리하고, 연구원은 등록/결재만 수행한다. 1차 문서군은 분석법/밸리데이션 doc/docx로 제한한다.

장점:

- 1차 범위는 좁지만 운영 구조는 실제 EDMS답게 간다.
- QA가 문서마다 수작업 확인하지 않고 시스템 설정으로 통제한다.
- 향후 SOP, 교육기록, 시험기록서 등으로 확장하기 좋다.

단점:

- 설정 화면과 권한 모델이 필요해 초기 구현량이 늘어난다.
- 전자서명 정책, 결재 라우트 예외, audit trail 설계를 처음부터 정확히 해야 한다.
- 내부 SOP 개정과 사용자 교육이 같이 필요하다.

### Approach C: 완전 EDMS Platform

처음부터 모든 문서군, doc/docx/xls/xlsx, 결재 템플릿, 배포/회수, 개정, 폐기, 교육 연계까지 포함한 플랫폼으로 설계한다.

장점:

- 장기 비전에 가장 가깝다.
- 여러 부서와 문서군을 한 번에 포괄할 수 있다.
- 상용 EDMS에 가까운 구조로 갈 수 있다.

단점:

- 첫 출시가 늦어진다.
- CSV와 Part 11 대응 범위가 너무 커진다.
- 분석법/밸리데이션 문서 흐름 검증 전에 범위가 폭발한다.

## Recommended Approach

Approach B: QA 설정 중심 EDMS Core를 선택한다.

전자결재/전자서명을 1차 범위에 포함하면서도, 문서군과 파일 형식을 분석법/밸리데이션 doc/docx로 제한해 실패 확률을 낮춘다. 동시에 QA가 수동 처리자가 아니라 시스템 통제 설정자 역할을 하도록 설계한다.

## Functional Scope

### Researcher

- 문서 등록 시작
- 과제코드 선택
- 문서종류 선택
- 시스템 자동 문서번호 확인
- doc/docx 업로드
- 작성자 전자서명
- 반려 시 수정본 재업로드
- 승인 상태 확인

### Reviewer / Approver

- 결재 대기 문서 확인
- 문서 원본 또는 변환 미리보기 확인
- 전자서명 전 재인증
- 승인, 반려, 의견 입력
- 서명 의미 확인: 작성, 검토, 승인 등

### QA / Document Controller

- 과제코드 관리
- 문서종류 관리
- 문서번호 규칙 관리
- 문서 종류별 결재 라우트 관리
- 권한 그룹 관리
- 워터마크 템플릿 관리
- 취소/반려/변환 실패 예외 처리
- audit trail 조회
- 배포 권한 및 열람 범위 관리

### Reader

- 권한이 있는 official PDF 검색
- 최신 effective 버전 열람
- 열람 이력 자동 기록
- 폐기/대체된 문서는 일반 열람에서 숨김 처리

## Document Lifecycle

- Reserved: 연구원이 등록을 시작해 문서번호가 발급됨
- Draft Uploaded: doc/docx 파일이 업로드됨
- In Review: 전자결재 진행 중
- Rejected: 반려됨. 번호는 재사용하지 않음
- Cancelled: 등록자가 취소함. 번호는 재사용하지 않음
- Approved: 모든 전자서명 완료
- Effective: official PDF 생성 완료, 공식 열람/배포본 활성화
- Superseded: 개정본으로 대체됨
- Retired: 더 이상 사용하지 않음

## Core Data Model

- Project: 과제코드, 과제명, 활성 여부
- Document Type: 문서종류 코드, 명칭, 번호 규칙, 결재 템플릿
- Document Number Sequence: 과제코드 + 문서종류별 일련번호 카운터
- Document: 문서번호, 제목, 과제코드, 문서종류, 상태, 현재 개정
- Document Revision: revision, 원본 파일, official PDF, 상태, effective date
- File Object: 파일명, 파일 종류, 저장 위치, 해시값, 크기, 업로드자, 업로드일시
- Approval Route: 문서종류별 결재 단계 템플릿
- Approval Task: 결재자, 단계, 상태, 의견, 완료일시
- Electronic Signature: 서명자, 서명 의미, 재인증 결과, 서명일시, 서명 대상 기록
- Watermark Template: 문서번호, revision, 상태, 출력일시, 열람자 표시 규칙
- Distribution Rule: 열람 가능 역할/부서/사용자
- View Event: 열람자, 문서번호, revision, PDF 식별자, 열람일시, IP/device 정보
- Audit Trail: 주요 행위, 이전값, 변경값, 행위자, 일시, 사유

## Electronic Signature Requirements

- 서명자는 고유 사용자 계정을 사용해야 한다.
- 서명 시 비밀번호 재입력 또는 2차 인증을 요구한다.
- 서명 의미를 명시한다: 작성, 검토, 승인.
- 서명은 특정 문서 revision과 파일 해시값에 연결된다.
- 서명 후 해당 revision의 원본 doc/docx는 변경 불가 상태가 된다.
- 서명 이력은 official PDF 또는 signature manifest에 연결된다.
- 반려 후 재업로드 시 이전 서명은 새 파일에 승계되지 않는다.

## Official PDF Generation

승인 완료 후 시스템은 doc/docx 전자 원본에서 official PDF를 생성한다.

필수 조건:

- 승인 완료된 revision의 원본 파일 해시값을 기록한다.
- PDF 변환 엔진과 버전을 기록한다.
- 변환 결과 PDF 해시값을 기록한다.
- 문서번호, revision, 상태, 생성일시 워터마크를 적용한다.
- official PDF는 변경 불가 저장소에 보관한다.
- 변환 실패 시 Effective 상태로 넘어가지 않는다.
- 변환 실패는 QA 예외 큐로 보낸다.

주요 리스크:

- doc/docx의 폰트, 표, 머리글/바닥글, 페이지 나눔이 PDF에서 달라질 수 있다.
- 변환 엔진이 바뀌면 같은 원본에서 다른 PDF가 나올 수 있다.
- 템플릿 표준화 없이 자동 PDF를 공식본으로 삼으면 품질 리스크가 크다.

완화:

- 1차 문서 템플릿을 표준화한다.
- 허용 폰트와 페이지 설정을 제한한다.
- 변환 엔진 버전을 고정하고 변경 시 영향평가를 수행한다.
- 샘플 문서 세트로 PDF 변환 검증을 수행한다.

## Audit Trail

감사추적 대상:

- 문서번호 자동 발급
- 파일 업로드, 교체, 삭제 시도
- 결재 라우트 생성과 변경
- 승인, 반려, 전자서명
- official PDF 생성
- 워터마크 적용
- Effective, Superseded, Retired 상태 변경
- 권한 변경
- official PDF 열람
- 다운로드 또는 출력 허용 시 해당 행위

Audit trail은 사용자가 수정할 수 없어야 하며, 시스템 관리자도 직접 변경할 수 없어야 한다.

## CSV / Part 11 Risk Areas

이 시스템은 전자기록과 전자서명을 다루므로 CSV와 Part 11 관점의 설계가 필요하다.

핵심 리스크:

- 전자서명이 실제 사용자와 연결되지 않는 경우
- 승인 후 파일이 바뀔 수 있는 경우
- official PDF와 승인된 원본 파일의 연결이 약한 경우
- audit trail이 누락되거나 수정 가능한 경우
- 권한 오류로 폐기본 또는 미승인본을 열람하는 경우
- 문서번호 취소/반려 이력이 사라지는 경우

필수 산출물:

- URS
- Functional Specification
- Risk Assessment
- Validation Plan
- IQ/OQ/PQ 또는 이에 준하는 검증 증적
- Audit Trail Review SOP
- Electronic Signature Policy
- Document Control SOP 개정
- Backup/Restore 검증
- Access Control Matrix

## Open Questions

- 문서종류별 승인 라우트는 몇 개의 대표 템플릿으로 시작할 수 있는가?
- 전자서명 시 비밀번호 재입력만으로 충분한가, 2차 인증이 필요한가?
- official PDF 다운로드를 허용할 것인가, 시스템 내 열람만 허용할 것인가?
- 출력본이 필요한 경우 출력 이력과 워터마크를 어떻게 관리할 것인가?
- 개정 문서가 Effective 될 때 이전본 회수/대체 확인이 필요한가?
- xls/xlsx 양식은 1차에서 등록만 할 것인가, 완전히 제외할 것인가?

## Success Criteria

1. 연구원이 분석법/밸리데이션 doc/docx 문서를 등록하면 시스템이 문서번호를 자동 발급한다.
2. 취소/반려된 문서번호가 재사용되지 않고 상태와 사유가 남는다.
3. 문서종류별 결재 라우트가 자동 적용된다.
4. 모든 승인자는 전자서명을 수행하고 서명 의미가 기록된다.
5. 승인 완료 후 official PDF가 자동 생성된다.

#### Core Engineering Findings

1. Critical: Document number generation needs concurrency control.
   Risk: Two researchers registering the same project/document type at the same time could receive duplicate or skipped numbers without a defensible audit reason.
   Fix: Use a database transaction with row-level lock or atomic sequence per `(project_code, document_type)`. Persist number reservation and audit event in the same transaction.

2. Critical: Audit trail must be append-only at the storage boundary, not just hidden in the UI.
   Risk: Admin or application bugs could alter regulated history.
   Fix: Use append-only audit events, hash chaining or tamper-evident digesting, strict DB permissions, and backup retention validation.

3. Critical: Electronic signature must bind to exact file versions.
   Risk: A signature over "document 123" is not enough if the uploaded file changes.
   Fix: Signature target must include document revision ID, source file hash, route step, signature meaning, signer identity, timestamp, and auth method.

4. High: PDF conversion is a regulated build artifact generator.
   Risk: The system could approve a file but produce a PDF with layout differences.
   Fix: Treat conversion engine as a validated component. Pin version, record version, test golden files, and route conversion failure to QA exception queue.

5. High: Authorization must check document state and revision, not just document ID.
   Risk: Users may open rejected, superseded, or retired versions.
   Fix: Viewer should resolve "current effective revision" by policy and require explicit privileged mode for historical versions.

6. High: Workflow configuration changes can invalidate in-flight approvals.
   Risk: A route template change mid-review could silently alter approval obligations.
   Fix: Snapshot the approval route onto each document revision at submission time. Later template changes affect only future revisions.

7. Medium: The plan does not define background job idempotency for PDF conversion.
   Risk: Retry after timeout could create multiple conflicting official PDFs.
   Fix: Use deterministic conversion job keys per revision and enforce one effective official PDF per revision.

8. Medium: Access review is missing.
   Risk: Permissions drift over time, especially after transfers or departures.
   Fix: Add periodic access review reports for QA/admin.
