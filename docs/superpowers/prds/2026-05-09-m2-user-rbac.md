# PRD: M2 — User Management & RBAC

> **Status:** ready-for-agent
> **Milestone:** M2
> **Date:** 2026-05-09
> **Author:** synthesized from URS/FS/DS/SOP-USER-001 + M1 codebase

---

## Problem Statement

After M1 Foundation, the EDMS system operates with a single hard-coded `admin` account. There is no mechanism to:

- Create new user accounts with appropriate roles
- Enforce role-based access: any authenticated user can access any endpoint
- Automatically expire AUDITOR accounts (GxP requirement: auditor independence expires periodically)
- Auto-unlock accounts locked after failed login attempts (locked accounts require manual DBA intervention)
- Generate a quarterly access-review report for QA sign-off (21 CFR Part 11 §11.10(d))

This violates FS Business Rules BR-USER-001 through BR-USER-012 and BR-ACC-001 through BR-ACC-004, and blocks M3 (Document & Numbering) which depends on user identity for authorship and e-signature.

---

## Solution

Implement a full User Administration layer with 3D Role-Based Access Control (Role × Document Category × Department). An `ADMIN` user gains REST API endpoints and a minimal admin UI to create, update, disable, and assign roles to users. Two automated schedulers enforce account validity: one auto-unlocks accounts 30 minutes after lockout, another disables and terminates sessions of expired AUDITOR accounts at midnight. A SQL view (`v_access_review`) and CSV export endpoint support the periodic access review SOP. All mutations write to the immutable `audit_logs` hash chain.

## User Stories

### Admin — User Lifecycle

1. As an ADMIN, I want to create a new user account with a full name, email, department, and initial role assignment, so that new staff can access the system without DBA involvement.
2. As an ADMIN, I want the system to generate a secure 16-character temporary password upon user creation, so that the new user must change it on first login (force_change_pw=true).
3. As an ADMIN, I want to receive the temporary password in-screen (or via email stub) immediately after user creation, so that I can securely hand it to the new employee.
4. As an ADMIN, I want to update a user's full name, email, department, title, and validity window (valid_from / valid_until), so that HR changes are reflected without re-creating the account.
5. As an ADMIN, I want to disable a user account, so that departing staff immediately lose access and any active sessions are terminated.
6. As an ADMIN, I want to reset a user's password, so that users who forget their password can regain access without DBA involvement.
7. As an ADMIN, I want to assign one or more system roles (AUTHOR, REVIEWER, APPROVER, QA, RA, READER, AUDITOR, ADMIN) to a user, so that the user's scope of actions is governed by their function.
8. As an ADMIN, I want to revoke a role from a user, so that users who change responsibilities lose associated privileges immediately (no session invalidation required — next request checks DB).
9. As an ADMIN, I want all user create/update/disable/role-change events to appear in the tamper-evident audit log, so that I can demonstrate GxP compliance to inspectors.

### Admin — Role & Permission Management

10. As an ADMIN, I want to view all 8 system roles and their descriptions, so that I can understand what access each role confers.
11. As an ADMIN, I want to update a role's display name and description, so that the UI reflects current organizational nomenclature without altering the immutable role code.
12. As an ADMIN, I want to grant a permission tuple (role, document category, department) so that a role holder can access a specific document category for a specific department.
13. As an ADMIN, I want to revoke a permission tuple, so that over-permissioned roles are corrected immediately.
14. As an ADMIN, I want deduplication of permission tuples enforced at the database level, so that I cannot accidentally create duplicate permission rows.
15. As an ADMIN, I want all role and permission changes audited, so that inspector review shows a complete chain of custody.

### Access Review

16. As a QA Manager, I want to download a CSV of all active users, their departments, their roles, and last login date, so that I can sign off on the quarterly access review (SOP-USER-001 §6.3).
17. As an AUDITOR, I want to receive a warning 30, 14, and 7 days before my account expires, so that I can request a renewal from the QA Manager without interruption.
18. As an AUDITOR, I want my account to be automatically disabled at midnight on my validity expiry date, so that auditor independence is maintained without manual intervention.
19. As an AUDITOR whose session was active when expiry ran, I want my session terminated immediately, so that I cannot continue operating on an expired credential.

### Account Self-Service & Automatic Recovery

20. As a locked-out user, I want my account to automatically unlock 30 minutes after lockout, so that I do not need to contact an administrator after a series of typos.
21. As a user, I want my login to be rejected if today is outside my valid_from–valid_until window, so that accounts are inactive before onboarding and after offboarding dates.
22. As a user, I want my last login date to be recorded on each successful authentication, so that the access review CSV reflects current activity.

### Security

23. As the system, I want the `/api/v1/admin/**` endpoint group to be accessible only to users with the `ADMIN` role, so that non-admin authenticated users are rejected with 403.
24. As the system, I want `@PreAuthorize` method-level security enabled, so that controllers and services can enforce role checks declaratively.
25. As the system, I want permission checks to be evaluated against the live database on every request (no cache), so that revoked access takes effect immediately per BR-ACC-004.
26. As the system, I want a `PermissionEvaluator` bean wired into Spring Security, so that future M3+ document-level `@PreAuthorize("hasPermission(doc, 'REVIEW')")` can be activated without structural changes.

### Developer / Validation

27. As a developer, I want OQ-USER-001 through OQ-USER-009 to be runnable as Testcontainers integration tests, so that each validation case produces machine-readable evidence for the GxP validation package.

## Implementation Decisions

### DB Layer (Flyway Migrations V6–V8)

- **V6 — `last_login_at`**: Add `TIMESTAMPTZ` column to `users` table. Mirror column in `users_aud` (Hibernate Envers tracks all user changes; missing column causes `ddl-auto: validate` failure).
- **V7 — `v_access_review` view**: Create or replace a PostgreSQL view joining `users`, `user_roles`, and `roles` tables. Aggregates role codes with `string_agg` per user. Includes last_login_at, department, full_name, status.
- **V8 — Permission table**: Create `permissions` table with `(role_id, category_id, department)` tuple. Use `UNIQUE NULLS NOT DISTINCT (role_id, category_id, department)` (PostgreSQL 15+) so an org-wide permission (NULL department) cannot coexist with a duplicate org-wide row. Add `permissions_aud` Envers mirror table.

### Permission Entity (new package: `permission`)

- `Permission` entity: `id`, `role` (FK → roles), `documentCategory` (FK → document_categories), `department` (nullable String, 50 chars), `createdAt`, `createdBy`.
- `DocumentCategory` entity: stub with `id`, `code` (NOT NULL UNIQUE), `name` (NOT NULL). Full CRUD deferred to M3.
- Envers `@Audited` on Permission. DocumentCategory is reference data — not audited individually.

### UserAdminService (new, `user` package)

Core decisions:
- `createUser` generates a 16-char secure-random temp password meeting the existing `PasswordPolicyValidator` rules (upper, lower, digit, special). Sets `force_change_pw=true`. Saves `PasswordHistory` entry so the user cannot re-use it on first forced change.
- `createUser` accepts a list of `roleCodes`; assigns roles transactionally in the same method.
- `disableUser` calls `SessionRegistry.getAllSessions(userId, false).forEach(SessionInformation::expireNow)` to terminate live sessions.
- `resetPassword` follows same flow as createUser password generation; also sets `force_change_pw=true`.
- All mutations call `AuditService.record(...)` within the service method (not in the controller).

### RoleAdminService (new, `user` package)

- Only `roleName` and `description` are mutable. `roleCode` and `isSystem` are immutable (enforced at service layer — not at DB layer). Attempting to change these fields is silently ignored (no exception) to allow partial-update DTOs.

### PermissionAdminService (new, `permission` package)

- `grantPermission` uses `saveAndFlush` to surface the `UNIQUE NULLS NOT DISTINCT` constraint as a `DataIntegrityViolationException`, which the controller maps to a `409 Conflict` ProblemDetail.

### AccessReviewService (new, `permission` package)

- Executes a native `SELECT * FROM v_access_review ORDER BY full_name` query via `EntityManager` (or `JdbcTemplate` on the main datasource). Streams rows into CSV format using Apache Commons CSV or manual StringBuilder (no new library dependency).

### Schedulers (new, `scheduler` package)

- `AccountUnlockScheduler`: `@Scheduled(fixedDelay = 60_000)`. Queries `users WHERE status = 'LOCKED' AND locked_at < NOW() - INTERVAL '30 minutes'`. Calls `AuditService` with action `ACCOUNT_AUTO_UNLOCKED`.
- `AuditorExpirySessionScheduler`: `@Scheduled(cron = "0 0 0 * * *", zone = "Asia/Seoul")`. Finds AUDITOR users where `valid_until < today`. Sets `status = DISABLED`. Expires sessions via `SessionRegistry`.
- `AuditorExpiryWarningScheduler`: `@Scheduled(cron = "0 5 0 * * *", zone = "Asia/Seoul")`. Finds AUDITOR users where `valid_until` is in {today+7, today+14, today+30}. Calls `EmailNotificationService.sendExpiryWarning(...)`.

### LocalAuthProvider Extension (modify)

- After password-match success, check `valid_from` and `valid_until`: reject if today is before `valid_from` or after `valid_until + 1 day` (inclusive: midnight on valid_until is still valid).
- On successful authenticate, update `user.lastLoginAt = OffsetDateTime.now()` and save.
- Emit `AuditService` record with action `USER_LAST_LOGIN_UPDATED`.

### PermissionEvaluator + Method Security (new, `config` package)

- `EdmsPermissionEvaluator implements PermissionEvaluator`:
  - `hasPermission(Authentication, Object, Object)` — M2 stub: return `false`.
  - `hasPermission(Authentication, Serializable, String, Object)` — M2 stub: return `false`.
- `MethodSecurityConfig extends GlobalMethodSecurityConfiguration` (or `@EnableMethodSecurity`): wires `EdmsPermissionEvaluator` into `DefaultMethodSecurityExpressionHandler`.
- SecurityConfig: add `.requestMatchers("/api/v1/admin/**").hasRole("ADMIN")` before the catch-all authenticated rule.

### AuditAction Enum Extension (modify)

Add: `USER_CREATED`, `USER_UPDATED`, `USER_DISABLED`, `USER_PASSWORD_RESET`, `ROLE_ASSIGNED`, `ROLE_REVOKED`, `ROLE_UPDATED`, `PERMISSION_GRANTED`, `PERMISSION_UPDATED`, `PERMISSION_REVOKED`, `ACCOUNT_AUTO_UNLOCKED`, `AUDITOR_EXPIRED`, `AUDITOR_EXPIRY_WARNING`, `USER_LAST_LOGIN_UPDATED`.

### REST API Contracts (new prefix `/api/v1/admin`)

| Method | Path | Purpose |
|--------|------|---------|
| GET | `/admin/users` | Paginated user list (page, size, search) |
| POST | `/admin/users` | Create user + assign roles |
| GET | `/admin/users/{userId}` | Single user detail |
| PUT | `/admin/users/{userId}` | Update user metadata |
| DELETE | `/admin/users/{userId}` | Disable user (soft delete) |
| POST | `/admin/users/{userId}/roles` | Assign role to user |
| DELETE | `/admin/users/{userId}/roles/{roleCode}` | Revoke role from user |
| POST | `/admin/users/{userId}/reset-password` | Issue new temp password |
| GET | `/admin/users/export?format=csv` | CSV access review export |
| GET | `/admin/roles` | List all roles |
| PUT | `/admin/roles/{roleId}` | Update role name/description |
| GET | `/admin/permissions` | List permissions (filter: roleId) |
| POST | `/admin/permissions` | Grant permission tuple |
| DELETE | `/admin/permissions/{permissionId}` | Revoke permission tuple |

All endpoints return `ProblemDetail` (RFC 7807) on error. Paginated list endpoints return a `PageResponse<T>` wrapper with `content`, `totalElements`, `totalPages`, `page`, `size`.

### Frontend Admin Views (Vue 3 / Pinia / inline styles)

Five new views added to existing Vue 3 / Vite app, following the `LoginView.vue` pattern (no UI library, inline styles):
- `UserListView.vue` — paginated table, search bar, create/disable buttons
- `UserCreateView.vue` — form: userId, fullName, email, department, roleCodes (multi-select)
- `UserEditView.vue` — update fields + role assignment panel
- `RoleListView.vue` — read-only list + edit-description inline
- `PermissionMatrixView.vue` — role × category × department matrix grid

New files: `src/api/adminApi.ts` (axios), `src/types/admin.ts` (TypeScript interfaces). Router entries added to `src/router/index.ts` behind `requiresAuth` guard.

### Email Stub

`EmailNotificationService` interface with single method `sendExpiryWarning(userId, validUntil, daysRemaining)`. `LogEmailNotificationService` implementation writes to SLF4J at WARN level. The `@Primary` bean is the log impl. M9 introduces `SmtpEmailNotificationService` via `@Profile("smtp")`.

## Testing Decisions

### What Makes a Good Test

Test external observable behavior, not internal implementation. Concretely:
- Assert the HTTP response status and body, not which service method was called.
- Assert the database state after a service call, not that a repository method was invoked.
- Assert audit log rows were written with correct action/actor/target, not that `AuditService.record(...)` was called.
- One test per behavior. Avoid shared mutable state between test cases.
- Use Testcontainers for integration tests — the project uses `@SpringBootTest @ActiveProfiles("test") @Import(TestcontainersConfig.class)` pattern established in M1. See `LocalAuthProviderIT`, `AuthServiceIT`, `SmokeIT` for prior art.

### Modules to Test (Integration Tests)

| Module | What to Test |
|--------|-------------|
| `UserAdminService` (via UserAdminController IT) | createUser → user row exists, password history row, force_change_pw=true, roles assigned, audit_logs entry with USER_CREATED + ROLE_ASSIGNED |
| `UserAdminService` — disableUser | status=DISABLED, active sessions expired, audit_logs entry |
| `RoleAdminService` (via RoleAdminController IT) | PUT /admin/roles/{id} updates roleName+description; roleCode unchanged |
| `PermissionAdminService` (via PermissionAdminController IT) | POST /admin/permissions → 201; duplicate → 409 |
| `AccessReviewService` | GET /admin/users/export?format=csv → Content-Type: text/csv, header row present, user rows present |
| `LocalAuthProvider` extension | valid_from future → 403 ACCOUNT_DISABLED; valid_until past → 403; success → last_login_at updated |
| `AccountUnlockScheduler` | schedule tick with locked user whose locked_at is >30 min ago → status=ACTIVE, audit log entry |
| `AuditorExpirySessionScheduler` | cron tick with expired AUDITOR → status=DISABLED, session expired |
| Security gate | unauthenticated /admin → 401; authenticated non-ADMIN → 403; ADMIN → 200 |
| `PermissionEvaluator` (unit) | hasRole("ADMIN") with ADMIN auth → true; hasRole("ADMIN") with non-ADMIN auth → false |

### Modules Not Requiring Dedicated ITs

- `EmailNotificationService` log stub — behavior verified by `AuditorExpiryWarningScheduler` IT.
- DTO record classes — no logic, no tests needed.
- `DocumentCategory` stub entity — covered by PermissionAdminController IT indirectly.
- Frontend views — manual smoke only (no automated UI tests in M2 scope).

## Out of Scope

- **Document-level permission evaluation** (`hasPermission(doc, 'REVIEW')`): PermissionEvaluator object/id forms are stubs returning `false`. Full evaluation activates in M3 when Document entity exists.
- **Custom (non-system) role creation**: Only the 8 seeded system roles exist. CRUD for custom roles is not in M2.
- **Email delivery**: `EmailNotificationService` is a log-only stub. SMTP wiring is M9.
- **LDAP / SSO auth provider**: Only `LOCAL` auth provider is implemented. External provider wiring is M10.
- **Document category CRUD**: `DocumentCategory` is a read-only seed entity in M2. Full CRUD is M3.
- **Two-person-rule (4-eyes)**: Role assignment by a single ADMIN is sufficient for M2. M7 may introduce approver workflow.
- **UI automated testing (Cypress / Playwright)**: Frontend admin views are smoke-tested manually only.
- **Password complexity UI feedback**: The change-password flow from M1 already enforces policy server-side. M2 adds no new UI complexity hints.

## Further Notes

- **GxP validation gate**: OQ-USER-001..009 (in `docs/validation/OQ_protocol.md`) must all pass as Testcontainers ITs before M2 can be declared complete. These tests constitute the Operational Qualification evidence for the user management subsystem.
- **Spec deviations confirmed at planning**: (1) `user_id` regex includes underscore per FS BR-USER-001; (2) `last_login_at` column is a DS §4.2.1 deviation; (3) `v_access_review` view is a DS §4.2.2 deviation; (4) `UNIQUE NULLS NOT DISTINCT` requires PostgreSQL 15+ (target is PG16, satisfied); (5) PUT /admin/roles only mutates roleName+description.
- **Session invalidation**: `SessionRegistry` is already wired in M1 SecurityConfig (`maximumSessions(1)` + `sessionRegistry` bean). `disableUser` and `AuditorExpirySessionScheduler` both use it to expire sessions immediately.
- **`audit_role` DB principal**: `AuditService` connects via the secondary `AuditDataSourceConfig` datasource. New audit actions added in M2 do not require changes to the datasource config — only the `AuditAction` enum and `AuditService.record(...)` call sites.
