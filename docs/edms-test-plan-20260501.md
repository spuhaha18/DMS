Generated by /autoplan on 2026-05-01.

## Scope

This test plan covers the first-release QA-controlled EDMS core:

- Analysis method/validation `doc/docx` documents
- Automatic document number generation
- Electronic approval and electronic signature
- Official PDF generation with watermark
- Controlled viewing and distribution
- View tracking and audit trail

## Test Strategy

Use a risk-based test strategy. The highest-risk flows are document number generation, electronic signature binding, official PDF generation, authorization, and audit trail integrity.

## Test Matrix

| Area | Test Type | Cases |
|---|---|---|
| Document number | Unit, integration, concurrency | Unique number per `(project_code, document_type)`, no reuse after cancel/reject, concurrent starts |
| File upload | Unit, integration | Accept doc/docx, reject unsupported files, hash stored, immutable source retained |
| Approval routing | Unit, integration | Correct route by document type, route snapshot per revision, approver order enforcement |
| Electronic signature | Integration, security | Reauth required, signature meaning captured, signature bound to revision and file hash |
| Rejection/resubmission | Integration | Old signatures invalidated, new revision or upload event created, audit trail complete |
| PDF conversion | Golden-file regression | Converter version recorded, watermark applied, PDF hash stored, conversion output matches expected samples |
| Conversion failure | Integration | No Effective transition, exception queue item created, retry logged |
| Authorization | Unit, integration, E2E | Only authorized roles can view current effective PDF, historical access privileged |
| Viewer tracking | E2E | Every view records user, document number, revision, timestamp, IP/device where available |
| Download/print exception | E2E, audit | Reason captured, user/time watermark applied, event logged |
| Audit trail | Unit, integration | Append-only events, before/after values, actor, timestamp, reason, query filters |
| Backup/restore | Operational | Restored system preserves files, hashes, signatures, audit trail, and effective status |
| Access lifecycle | Integration | Disabled users cannot sign/view, role changes affect future access, access review report generated |

## Golden File Set

Before implementation, collect at least 5 representative `doc/docx` files:

1. Simple text-heavy validation document
2. Table-heavy analysis method document
3. Header/footer-heavy document
4. Document with embedded image or diagram
5. Long document with section/page breaks

For each file, capture expected PDF output and known layout risks.

## Release Gate

The first release cannot be accepted until:

- No duplicate document number can be produced in concurrency tests.
- Every electronic signature is bound to a specific revision and source file hash.
- PDF conversion passes the golden-file set.
- Effective state cannot be reached without approved route, official PDF hash, and audit trail events.
- Unauthorized users cannot view official PDFs.
- Audit trail survives backup/restore verification.
