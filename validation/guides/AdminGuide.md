## 1. 사용자 관리

사용자 관리는 SOP-USER-001과 연계된다. 모든 변경은 계정 대장(SOP-USER-001 부록 B)에 동시에 업데이트해야 한다.

### 1.1 신규 사용자 생성

1. **[Admin]** → **[사용자 관리]** → **[신규 생성]** 클릭
2. 계정 요청 양식(SOP-USER-001 부록 A)을 참조하여 아래 항목을 입력:

| 항목 | 입력 내용 | 비고 |
|---|---|---|
| 사용자 ID | 영문·숫자·점·하이픈 조합, 2~50자 | 중복 불가, 생성 후 변경 불가 |
| 이름 | 실명 (한글) | |
| 이메일 | 회사 이메일 | 중복 불가 |
| 부서 | 소속 부서 | |
| 직위 | 직책 | |
| 역할 | 1개 이상 선택 | Author/Reviewer/Approver/QA/RA/Reader/Admin/Auditor |
| 유효기간 | Auditor 계정만 설정 | valid_from, valid_until |

3. 초기 비밀번호 설정 (임시 비밀번호 규칙: 대소문자+숫자+특수문자, 12자 이상)
4. **[저장]** → 계정 생성 완료. `force_change_pw = true` 자동 설정.
5. 생성된 계정 정보를 본인에게 개인 이메일로 발송. 임시 비밀번호는 별도 전화·방문으로도 전달.
6. 계정 대장 업데이트.

> 계정 생성 전 교육 이수 여부를 확인하세요 (SOP-TRAINING-001).

### 1.2 역할 변경

1. **[Admin]** → **[사용자 관리]** → 대상 사용자 검색 → **[수정]**
2. 역할 탭에서 기존 역할 제거/추가
3. 변경 사유를 감사로그 메모에 입력 (EDMS 자동 기록)
4. 계정 대장 업데이트

### 1.3 계정 비활성화

**원칙**: EDMS에서 계정은 삭제하지 않는다. 비활성화만 수행.

1. **[Admin]** → **[사용자 관리]** → 대상 사용자 검색 → **[비활성화]**
2. 비활성화 사유 입력 (필수)
3. **[확인]** → 즉시 처리:
   - 계정 status = DISABLED
   - 활성 세션 즉시 만료
   - 감사로그 자동 기록
4. 계정 대장 업데이트 (비활성화 일자, 사유 기록)

퇴직자 비활성화는 **마지막 근무일 종료 전**에 처리해야 한다 (SOP-USER-001 §5.3).

### 1.4 비밀번호 초기화

1. **[Admin]** → **[사용자 관리]** → 대상 사용자 → **[비밀번호 초기화]**
2. 임시 비밀번호 생성 (시스템 자동 생성 또는 관리자 입력)
3. 사용자 이메일로 자동 발송 + 구두로도 전달
4. `force_change_pw = true` 자동 설정 → 다음 로그인 시 강제 변경

### 1.5 잠금 해제

5회 오입력으로 잠긴 계정:
1. **[Admin]** → **[사용자 관리]** → 잠긴 계정 검색 (status = LOCKED)
2. **[잠금 해제]** 버튼 클릭
3. 필요 시 비밀번호 초기화 병행

## 2. 권한 매트릭스 설정

권한은 **역할 × 문서카테고리 × 부서** 3차원으로 설정한다. 모든 권한은 최소 권한 원칙(Least Privilege)을 따른다.

### 2.1 권한 조회

**[Admin]** → **[권한 관리]** → 역할 또는 카테고리 필터로 현재 권한 확인

각 권한 항목:
| 권한 | 설명 |
|---|---|
| can_view | 문서 목록·상세 조회 |
| can_download | PDF 다운로드 |
| can_create | 문서·버전 생성 |
| can_edit_draft | 초안 파일 업로드·수정 |
| can_review | 검토 서명 (REVIEWED) |
| can_approve | 승인 서명 (APPROVED/QA_APPROVED) |
| can_retire | 문서 폐기 |

### 2.2 권한 추가/수정

1. **[Admin]** → **[권한 관리]** → **[권한 추가]**
2. 역할, 카테고리, 부서(전 부서는 "전체" 선택) 지정
3. 필요한 권한 체크박스 선택
4. **[저장]** → 즉시 적용 (세션 재로그인 불필요)

> 예시 — QC 부서 Author에게 SOP 작성·검토 요청 권한:
> - 역할: AUTHOR, 카테고리: SOP, 부서: QC
> - can_view ✓, can_create ✓, can_edit_draft ✓

### 2.3 권한 삭제

권한 목록에서 **[삭제]** 버튼 → 즉시 적용. 해당 권한 보유 사용자의 접근이 즉시 차단된다.

### 2.4 기밀 문서(confidential) 주의

기밀 문서는 추가 격리 필요. 기밀 문서 접근 허용은 Admin이 별도로 해당 사용자에게만 명시적 권한 설정.

## 3. 워크플로 템플릿 관리

워크플로 템플릿은 카테고리별로 하나씩 설정한다. 변경 전 SOP-CHANGE-001 Minor 변경 통제를 거쳐야 한다.

### 3.1 템플릿 조회

**[Admin]** → **[워크플로 템플릿]** → 카테고리별 템플릿 목록

### 3.2 단계 설정

각 단계(Step)의 설정 항목:

| 항목 | 설명 |
|---|---|
| 순서 | 1부터 시작하는 단계 번호 |
| 단계 유형 | REVIEW (검토) 또는 APPROVAL (승인) |
| 역할 | 해당 단계를 수행하는 역할 코드 |
| 최소 서명자 수 | 단계 완료에 필요한 최소 서명 수 |
| 병렬 수행 | 동일 단계에서 복수 서명자가 동시에 서명 가능 여부 |
| 자동 배정 | 해당 역할 보유 사용자에게 자동 워크플로 배정 여부 |
| QA 필요 | 이 단계가 QA 역할 서명을 필수로 요구하는지 여부 |

### 3.3 카테고리 qa_mandatory 설정

카테고리의 `qa_mandatory = true`이면, 워크플로 전 과정에서 QA 역할의 서명이 반드시 포함되어야 한다. **[Admin]** → **[카테고리 관리]** → 해당 카테고리 → `QA 승인 필수` 체크박스.

## 4. 채번 템플릿 관리

문서 번호는 시스템이 자동 채번한다. 임의 변경 불가.

### 4.1 템플릿 설정

**[Admin]** → **[채번 템플릿]** → 카테고리별 설정

**형식 패턴 플레이스홀더**:
| 플레이스홀더 | 치환 내용 | 예시 |
|---|---|---|
| `{TYPE}` | 카테고리 코드 | SOP, METHOD |
| `{DEPT}` | 부서 코드 | QC, RD |
| `{PROD}` | 제품 코드 | PRJ001 |
| `{YEAR}` | 4자리 연도 | 2026 |
| `{SEQ:N}` | N자리 일련번호 | {SEQ:3} → 001 |

**카운터 범위**:
| 범위 | 설명 |
|---|---|
| PER_DEPT | 부서별로 일련번호 재시작 |
| PER_PRODUCT | 제품/프로젝트별 재시작 |
| PER_YEAR | 연도별 재시작 (1월 1일) |
| GLOBAL | 전체 누적 (재시작 없음) |

예시:
- `{TYPE}-{DEPT}-{SEQ:3}` + `PER_DEPT` → SOP-QC-001, SOP-QC-002 ...
- `{TYPE}-{DEPT}-{YEAR}-{SEQ:3}` + `PER_YEAR` → SOP-QC-2026-001

### 4.2 채번 미리보기

설정 변경 전 **[미리보기]** 기능으로 실제 채번 없이 결과를 확인할 수 있다.

## 5. 카테고리 및 정기검토 주기 설정

**[Admin]** → **[카테고리 관리]**

| 설정 항목 | 설명 | 기본값 |
|---|---|---|
| 카테고리 코드 | SOP / METHOD / SPEC / FORM (변경 불가) | |
| 카테고리명 | 한글 표시명 | |
| 정기검토 주기 | 개월 수 (예: 24 = 2년) | 24 |
| QA 승인 필수 | 카테고리 전체에 QA 서명 필수 여부 | false |
| 활성 여부 | 비활성화 시 해당 카테고리 신규 문서 생성 불가 | true |

정기검토 알림 발송 시점 (매일 00:10 KST 자동 실행):
- `effective_date + review_period_months - 90일 / 30일 / 7일 / 0일`

## 6. 감사로그 조회 및 내보내기

### 6.1 감사로그 조회

**[감사]** → **[감사로그]**

필터 옵션:
- 사용자 ID 또는 성명
- 액션 유형 (예: DOCUMENT_SUBMITTED, USER_DISABLED 등)
- 엔티티 유형 (DOCUMENT_VERSION, USER 등)
- 기간 (시작일~종료일)

조회 결과에서 해시(`this_hash`)를 클릭하면 해시체인 연속성을 확인할 수 있다.

### 6.2 감사로그 CSV 내보내기

1. **[감사]** → **[감사로그]** → 기간 필터 설정
2. **[CSV 내보내기]** 버튼 클릭
3. 파일 저장 (감사인 제출, 내부 분석용)

### 6.3 해시체인 무결성 검증

**[감사]** → **[무결성 검증]** → 기간 입력 → **[검증 실행]**

결과:
- `"valid": true` — 해당 기간 감사로그 해시체인 정상
- `"valid": false` — 단절 ID 표시 → 즉시 SOP-INCIDENT-001 발동

### 6.4 일별 WORM 앵커 확인

**[감사]** → **[WORM 앵커]** → 날짜별 Merkle root, 레코드 수, MinIO 저장 경로 확인

## 7. 백업·복구 운영

상세 절차는 SOP-BACKUP-001을 참조한다. 여기서는 일상 모니터링 포인트만 기술한다.

### 7.1 일일 백업 상태 확인 (매일 아침)

```bash
# PostgreSQL 백업 최근 실행 시각 확인
ls -la /backup/pg/ | tail -5

# WAL 아카이빙 상태 확인 (마지막 수신 시각)
tail -20 /var/log/pg_archive.log
```

백업이 실패한 경우 → 시스템 관리자 즉시 원인 파악 + QA 보고 (SOP-DEVIATION-001 Major)

### 7.2 MinIO Replication 상태 확인

```bash
# Site Replication 상태
mc admin replicate info primary/
```

### 7.3 반기 복구 훈련 (SOP-BACKUP-001 §5.4)

매년 2회 (1월, 7월) DR 환경에서 복구 훈련을 실시한다. 훈련 결과를 SOP-BACKUP-001 부록 B 복구 훈련 기록서에 기록하고 QA 서명.

## 8. 모니터링 대시보드

### 8.1 Prometheus + Grafana

운영 서버에서 `https://monitoring.lab.internal/grafana`로 접속 (Admin 계정):

주요 모니터링 지표:

| 지표 | 정상 범위 | 알림 임계값 |
|---|---|---|
| API 응답 시간 (P95) | < 2초 | > 5초 |
| DB 연결 풀 사용률 | < 80% | > 90% |
| MinIO 디스크 사용률 | < 70% | > 85% |
| PDF 변환 큐 대기 | < 10건 | > 50건 |
| SSE 연결 수 | 활성 사용자 수와 유사 | 비정상 급증 |
| JVM 힙 메모리 사용률 | < 80% | > 90% |
| audit_logs INSERT 실패 | 0건 | 1건 이상 |

### 8.2 로그 조회 (운영 서버)

```bash
# Spring Boot 애플리케이션 로그
journalctl -u edms -n 200 --no-pager

# 감사 전용 로그 (별도 파일로 분리됨)
tail -100 /var/log/edms/audit.log

# Nginx 접근 로그
tail -100 /var/log/nginx/access.log

# NTP 동기화 상태
chronyc tracking
```

### 8.3 일상 점검 항목 (매일)

- [ ] 배치 작업 정상 실행 여부 (01:00 KST — WORM 앵커 생성, 00:10 KST — 정기검토 스케줄러)
- [ ] 백업 정상 완료 여부
- [ ] Prometheus 알림 무시된 항목 없는지 확인
- [ ] 전날 비업무 시간 이상 이벤트 없는지 확인 (SOP-AUDIT-TRAIL-001 §5.1 자동 알림)

## 9. 분기 계정 접근 검토

**주기**: 분기 1회  
**수행자**: 시스템 관리자 + QA  
**근거**: SOP-USER-001 §5.4

### 9.1 활성 계정 목록 내보내기

**[Admin]** → **[사용자 관리]** → 필터: status = ACTIVE → **[CSV 내보내기]**

### 9.2 검토 항목

| 검토 항목 | 방법 |
|---|---|
| 퇴직·이동 계정 | 내보낸 CSV를 HR 시스템과 대조. 비활성화 누락 확인 |
| 30일 이상 미로그인 | 사용자 목록에서 `last_login_at < NOW() - 30days` 필터 |
| 역할-직무 적정성 | 현재 직무 대비 과다 권한 보유 계정 |
| Auditor 유효기간 | `valid_until < NOW() + 7days` 계정 확인 |
| Admin 역할 계정 수 | Admin 계정은 2인 이하 권장. 현황 확인 |

### 9.3 조치 및 기록

1. 비활성화 필요 계정 → SOP-USER-001 §5.3 절차 실행
2. 역할 변경 필요 계정 → 부서 책임자 확인 후 SOP-USER-001 §5.2.1 절차 실행
3. 검토 결과를 SOP-USER-001 부록 C 분기 계정 검토 기록서에 기록
4. 시스템 관리자·QA 공동 서명

## 10. Admin 역할 사용 시 주의사항

Admin 권한은 강력하므로 아래 원칙을 반드시 준수한다:

1. **업무 이외 목적으로 Admin 권한 사용 금지**
2. **Admin 계정으로의 직접 문서 생성·서명 지양** (일반 역할 계정 사용 권장)
3. **모든 Admin 행위는 audit_logs에 기록**됨 (SOP-AUDIT-TRAIL-001 §5.3에서 100% 검토)
4. **비밀번호 관리 강화**: Admin 비밀번호는 14자 이상, 타인과 절대 미공유
5. **Admin 공유 계정 금지**: 2인이 각각 개인 Admin 계정 보유 (공유 Admin 계정 사용 금지)
