## 1. 목적

본 SOP는 EDMS 사용자 계정의 생성·수정·비활성화·정기 검토 절차를 정의한다.

21 CFR Part 11 §11.10(d) (시스템 접근 제한) 및 §11.300 (암호 관리) 요건을 충족하며, 비인가 접근을 방지하고 각 사용자의 행위를 감사추적으로 식별 가능하게 유지한다.

---

## 2. 범위

본 SOP는 EDMS에 접근하는 모든 사용자(내부 직원, 외부 감사인 포함)에게 적용된다.

---

## 3. 용어 정의

| 용어 | 정의 |
|---|---|
| 역할 (Role) | 시스템 내 기능 권한 집합 (Author, Reviewer, Approver, QA, RA, Reader, Admin, Auditor) |
| 권한 매트릭스 | 역할 × 문서카테고리 × 부서 범위로 정의된 접근 통제 설정 |
| 계정 비활성화 | 계정을 삭제하지 않고 로그인 불가 상태(DISABLED)로 전환하는 행위 |
| 계정 유효기간 | Auditor 역할에 설정하는 접근 허용 기간 |
| 비밀번호 초기화 | Admin이 사용자의 비밀번호를 임시값으로 재설정하는 행위 |

---

## 4. 책임

| 역할 | 책임 |
|---|---|
| 부서 책임자 | 신규 계정 요청, 역할 변경 요청, 퇴직·이동 시 계정 처리 요청 |
| 시스템 관리자(Admin) | 계정 생성·수정·비활성화 실행, 비밀번호 초기화, 분기 접근 검토 수행 |
| QA | 연 1회 계정 접근 검토 참여, 역할-직무 적정성 확인 |
| 사용자 본인 | 비밀번호 보안 관리, 계정 공유 금지, 이상 징후 즉시 신고 |

---

## 5. 절차

### 5.1 신규 계정 생성

#### 5.1.1 계정 생성 요청

1. 부서 책임자가 **계정 요청 양식(부록 A)** 을 작성하여 시스템 관리자에게 제출한다.
   - 필수 항목: 이름, 이메일, 부서, 직위, 요청 역할, 문서 접근 범위(카테고리·부서)
   - Auditor 계정: 유효기간(시작일, 종료일) 및 방문 목적 필수 기입
2. 시스템 관리자는 요청 수신 후 **1영업일 이내** 처리한다.

#### 5.1.2 계정 생성 실행

1. EDMS Admin 화면 → [사용자 관리] → [신규 생성] 클릭
2. 아래 정보를 입력한다.

| 항목 | 입력 내용 |
|---|---|
| 사용자 ID | 영문·숫자·점·하이픈 조합, 2~50자 |
| 이름 | 한글 실명 |
| 이메일 | 회사 이메일 주소 |
| 부서 | 소속 부서 |
| 직위 | 직책 |
| 역할 | 요청 양식에 명시된 역할 (1개 이상) |
| 유효기간 | Auditor 계정에만 설정 |

3. 초기 비밀번호를 설정한다 (임시 비밀번호 규칙: 영문 대소문자+숫자+특수문자 12자 이상).
4. [저장] 클릭 → 계정 생성 완료 (자동으로 비밀번호 변경 강제 플래그 설정됨).
5. 생성된 계정 정보(사용자 ID, 임시 비밀번호, 로그인 URL)를 본인에게 **개인 이메일**로 발송한다. 임시 비밀번호는 이메일 전송 후 별도 경로(전화 또는 직접 구두)로도 알린다.
6. **계정 대장(부록 B)** 에 등록한다.

#### 5.1.3 권한 매트릭스 설정

역할 배정만으로는 문서 접근이 허용되지 않는다. Admin 화면 → [권한 매트릭스]에서 아래를 설정한다.

- 역할 × 문서카테고리 × 부서의 조합으로 구체적인 접근 권한 체크

> **원칙**: 최소 권한 원칙(Least Privilege) — 업무 수행에 필요한 최소 권한만 부여한다.

---

### 5.2 계정 정보 수정

#### 5.2.1 역할 변경

1. 부서 책임자가 역할 변경 요청 (부록 A 양식의 [변경 유형: 역할 변경] 선택).
2. 시스템 관리자가 EDMS Admin → [사용자 관리] → 해당 계정 → 역할 수정.
3. 변경 사유를 감사로그에 기록한다 (EDMS가 자동 기록).
4. 계정 대장을 업데이트한다.

#### 5.2.2 부서 이동

1. 부서 책임자(이동 전·후 모두)가 계정 수정 요청.
2. 부서 정보 및 접근 권한 매트릭스를 새 부서 기준으로 업데이트.
3. 기존 부서 문서에 대한 접근 권한 제거 여부를 확인한다 (업무 인수인계 완료 후).

#### 5.2.3 비밀번호 초기화

사용자가 비밀번호를 잊은 경우:
1. 사용자가 본인 신원 확인 후 시스템 관리자에게 요청 (직접 방문 또는 부서 책임자 통해).
2. 시스템 관리자가 Admin → [비밀번호 초기화] 실행.
3. 임시 비밀번호를 본인에게 개인 이메일로 발송 (비밀번호 변경 강제 플래그 자동 설정).

---

### 5.3 계정 비활성화

> **원칙**: EDMS에서 계정은 삭제하지 않는다. 비활성화만 수행하며, 과거 서명·감사로그 등 모든 이력이 보존된다.

#### 5.3.1 비활성화 트리거

| 사유 | 처리 기한 |
|---|---|
| 퇴직 | 마지막 근무일 종료 전 |
| 장기 부재 (30일 이상 미사용) | 부서 책임자 요청 후 1영업일 이내 |
| 보안 사고 (계정 침해 의심) | 즉시 |
| Auditor 유효기간 만료 | 시스템 자동 처리 (별도 수동 조치 불필요) |
| 분기 검토에서 불필요 계정 확인 | 검토 완료 후 5영업일 이내 |

#### 5.3.2 비활성화 절차

1. 부서 책임자가 비활성화 요청 (부록 A 양식의 [변경 유형: 비활성화] 선택, 사유 필수 기입).
2. 시스템 관리자가 EDMS Admin → [사용자 관리] → 해당 계정 → [비활성화] 클릭.
3. 비활성화 사유 입력 → 확인 (감사로그 자동 기록).
4. 해당 사용자의 활성 세션이 즉시 만료됨을 확인한다 (EDMS 자동 처리).
5. 계정 대장을 업데이트한다 (비활성화 일자, 사유 기록).

---

### 5.4 분기 계정 접근 검토

**주기**: 분기 1회  
**수행자**: 시스템 관리자 (QA 참여)  
**목적**: 비인가 계정, 과다 권한 계정, 장기 미사용 계정을 주기적으로 확인하고 정리한다.

#### 5.4.1 검토 절차

1. EDMS에서 전체 활성 계정 목록을 내보낸다 (Admin → [사용자 관리] → CSV 내보내기).
2. 아래 항목을 검토한다.

| 검토 항목 | 확인 내용 |
|---|---|
| 퇴직·이동 계정 | HR 시스템과 대조하여 비활성화 누락 계정 확인 |
| 30일 이상 미로그인 | 미사용 계정 비활성화 필요 여부 부서 책임자 확인 |
| 역할-직무 적정성 | 현 직무 대비 과다 권한 계정 확인 |
| Auditor 유효기간 | 만료 임박(7일 이내) 계정 사전 연장 또는 만료 처리 |
| Admin 역할 계정 수 | Admin 계정은 최소화 원칙 적용 (2인 이하 권장) |

3. 조치가 필요한 계정은 부서 책임자 확인 후 처리한다.
4. 검토 결과를 **계정 검토 기록서(부록 C)** 에 기록하고 시스템 관리자와 QA가 서명한다.

---

### 5.5 비밀번호 관리 원칙

사용자는 아래 규칙을 준수해야 한다.

| 규칙 | 내용 |
|---|---|
| 비밀번호 최소 길이 | 8자 이상 |
| 복잡도 | 대문자·소문자·숫자·특수문자 중 3종 이상 |
| 이전 비밀번호 재사용 | 최근 5개 비밀번호와 동일 불가 |
| 계정 공유 | **금지** — 개인 계정을 타인과 공유하지 않는다 |
| 임시 비밀번호 변경 | 최초 로그인 시 즉시 변경 (시스템 강제) |
| 비밀번호 유출 의심 시 | 즉시 시스템 관리자에게 신고 → 비밀번호 초기화 |

> **계정 공유 금지 근거**: 21 CFR Part 11 §11.100 — 전자서명은 서명자 개인을 고유하게 식별해야 한다. 계정 공유 시 서명의 법적 효력이 소멸될 수 있다.

---

### 5.6 계정 이상 징후 대응

| 이상 징후 | 1차 조치 | 보안사고 에스컬레이션 기준 |
|---|---|---|
| 비업무 시간(22:00~06:00) 로그인 감지 | audit_logs 확인 → 해당 사용자 확인 → 필요 시 계정 잠금 | 사용자 본인이 아님 확인 시 → **SOP-INCIDENT-001** (Account Compromise) 발동 |
| 단기간 과다 서명 (1시간 내 10건 이상) | audit_logs 확인 → 해당 사용자 확인 | 의도하지 않은 서명 확인 시 → **SOP-INCIDENT-001** (Account Compromise / Tampering) 발동 |
| 계정 잠금 빈번 발생 (주 3회 이상) | 사용자 면담 → 즉시 비활성화 | 피싱·계정 탈취 의심 시 → **SOP-INCIDENT-001** (Account Compromise) 발동 |
| 비인가 IP에서의 접근 시도 | 방화벽 로그 확인 → IT 보안팀 보고 | 실제 침투 확인 시 → **SOP-INCIDENT-001** (Unauthorized Access) 발동 |

> **에스컬레이션 원칙**: 단순 이상 징후는 본 절차(§5.6) 내에서 처리한다. 계정 침해·비인가 접근·데이터 위변조 의심이 있는 경우 **즉시 SOP-INCIDENT-001**을 발동하며, 두 절차는 동시에 진행될 수 있다.
