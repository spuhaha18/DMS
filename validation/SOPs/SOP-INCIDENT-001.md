## 1. 목적

본 SOP는 EDMS에서 발생하는 보안사고(계정 침해, 비인가 접근, 데이터 위변조, 시스템 침투 등)를 신속하게 탐지하고, 봉쇄하고, 복구하는 절차를 정의한다. EU GMP Annex 11 §13 및 Part 11 §11.10(d)(e)의 보안 요건에 부합한다.

---

## 2. 범위

본 SOP는 EDMS 운영 환경에서 발생하는 다음 유형의 보안사고에 적용된다. 단순 시스템 오류·절차 위반은 SOP-DEVIATION-001 적용.

| 사고 유형 | 설명 |
|---|---|
| **Privilege Escalation** | 권한 없는 Admin 또는 고권한 역할 획득 |
| **Account Compromise** | 타인의 계정을 탈취하여 부정 접근·서명 |
| **Unauthorized Access** | 접근 권한 없는 문서·기능에 대한 접근 |
| **Data Exfiltration** | GxP 데이터의 무단 외부 유출 |
| **Tampering** | 전자기록·감사추적·서명 위변조 시도 또는 완료 |
| **DoS/System Disruption** | 시스템 가용성을 의도적으로 저해하는 행위 |
| **Insider Threat** | 내부자에 의한 의도적 보안 위반 |

---

## 3. 용어 정의

| 용어 | 정의 |
|---|---|
| 탐지 (Detection) | 이상 이벤트가 보안사고임을 인지하는 시점 |
| 봉쇄 (Containment) | 사고가 추가 확산되는 것을 막는 즉각 조치 |
| 박멸 (Eradication) | 사고의 근본 원인 제거 (취약점 패치, 계정 폐기 등) |
| 복구 (Recovery) | 정상 운영 상태로 복원 |
| 사후 분석 (Lessons Learned) | 사고 원인·대응 분석 및 재발 방지 조치 |

---

## 4. 책임

| 역할 | 책임 |
|---|---|
| 전체 사용자 | 이상 징후 즉시 신고 |
| 시스템 관리자 | 탐지·봉쇄·복구 실행, 24시간 이내 내부 보고 |
| QA | 사고 조사 참여, 데이터 무결성 재검증, 보고서 승인 |
| QA Manager | 외부 보고 결정, 최종 보고서 승인 |
| 보안팀 (IT) | 기술적 침해 분석, 방화벽·로그 분석 |
| 부서 책임자 | 해당 부서 사용자 행동 확인 |

---

## 5. 절차

### 5.1 탐지 (Detection)

보안사고는 다음 경로로 탐지된다:

**자동 탐지 (시스템)**:
- 비업무 시간(22:00~06:00) 로그인 → 이메일 alert
- 1시간 내 10건 이상 서명 → 이메일 alert
- 5회 로그인 실패 → 계정 잠금 + alert
- 감사추적 해시 단절 → 즉시 CRITICAL alert (SOP-AUDIT-TRAIL-001 §5.1)
- 비인가 IP에서 접근 시도 → 방화벽 로그

**수동 탐지 (사용자·관리자)**:
- 사용자 신고: 계정 무단 접근 의심, 비밀번호 유출 의심
- 관리자 SOP-AUDIT-TRAIL-001 §5.2~5.3 주별·월별 검토 중 이상 발견
- 시스템 로그 이상 발견

### 5.2 초기 평가 및 분류 (30분 이내)

탐지 후 30분 이내에 시스템 관리자가 초기 평가를 수행한다:

| 확인 항목 | 방법 |
|---|---|
| 사고 유형 확인 | §2 분류 기준 적용 |
| 영향 범위 추정 | audit_logs 최근 이벤트 조회 (관련 actor_user_id, entity_id) |
| 현재 활성 세션 확인 | Spring Session 저장소 조회 |
| 최근 서명 유효성 | 관련 signature_manifests 조회 |

분류 결과에 따라 에스컬레이션:
- **Critical**: QA Manager 즉시 전화 통보
- **Major**: QA에 이메일 + 메신저 통보
- **Minor**: 이메일로 보고

### 5.3 봉쇄 (Containment)

**즉시 조치 (탐지 후 1시간 이내)**:

| 상황 | 조치 |
|---|---|
| 계정 침해 의심 | 해당 계정 즉시 비활성화(DISABLED) + 활성 세션 강제 만료 |
| 권한 오용 의심 | 해당 계정 역할 즉시 제거 |
| 광범위 침해 의심 | Admin이 해당 부서/역할 전체 세션 만료 처리 |
| 데이터 위변조 의심 | 영향받은 문서 버전 DRAFT 이전으로 격리 (상태 변경 중단) |
| 네트워크 침해 의심 | 보안팀 + 방화벽 담당자 즉시 연락, 필요 시 앱 서버 네트워크 격리 |

**증거 보존**:
```sql
-- 관련 계정 최근 행위 전체 조회 (변경 전 스냅샷)
SELECT * FROM audit_logs
WHERE actor_user_id = '<suspect_user>'
  AND server_ts >= NOW() - INTERVAL '30 days'
ORDER BY server_ts;

-- 관련 세션 정보
SELECT * FROM spring_session WHERE principal_name = '<suspect_user>';
```

MinIO 버킷 조회 기록 (접근 로그가 있는 경우)도 보존한다. 증거를 삭제하거나 수정하지 않는다.

### 5.4 박멸 (Eradication)

근본 원인 확인 후 다음을 실행한다:

| 원인 | 조치 |
|---|---|
| 취약한 비밀번호·탈취 | 시크릿 전체 회전 (VAL-SEC-001 §6 참조), 비밀번호 정책 강화 검토 |
| 소프트웨어 취약점 | SOP-CHANGE-001 Emergency 경로로 패치 배포 (SOP-CICD-001 §5.5) |
| 내부자 계정 오용 | 계정 영구 비활성화, 법무팀/HR 연계 검토 |
| 방화벽 우회 | 보안팀 방화벽 룰 긴급 변경 + 전수 감사 |
| 물리적 침해 | 물리 보안팀 조치, 서버 포렌식 |

### 5.5 복구 (Recovery)

1. 영향받은 GxP 데이터의 무결성 재검증:
   ```bash
   # API를 통한 해시체인 무결성 검증
   POST /api/v1/audit-logs/checkpoints/verify
   { "from_date": "<사고 시작 추정일>", "to_date": "<현재>" }
   ```
2. 영향받은 전자서명의 유효성 QA 확인
3. 서비스 정상 복구 확인 (smoke test: 로그인, 문서 조회, 서명 기능)
4. 격리했던 계정·세션 복구 (사고 원인 제거 확인 후)
5. 영향받은 기록의 처리 방향 QA 결정:
   - 서명이 위조된 경우: 해당 버전 RETIRED 처리 + QA 사유 기록
   - 서명이 유효한 경우: 감사기록에 사고·조사 사실 메모 추가

### 5.6 보고 (Reporting)

**내부 보고**:
- 24시간 이내: 시스템 관리자 → QA Manager 서면 보고 (부록 A 양식)
- 72시간 이내: QA Manager → 경영진 보고 (Critical 사고)

**외부 보고 의무** (해당 시):
| 해당 상황 | 보고 대상 | 법정 기한 |
|---|---|---|
| 개인정보 유출 (임직원·감사인 정보) | 개인정보보호위원회 | 72시간 이내 |
| GMP 데이터 위변조로 제품 품질 영향 | 식품의약품안전처 | 해당 규정 기한 |
| 국가핵심기술 관련 데이터 유출 (해당 시) | 산업통상자원부 | 즉시 |

외부 보고 여부는 QA Manager + 법무팀이 공동 결정한다.

### 5.7 사후 분석 (Lessons Learned)

사고 종결 후 5영업일 이내:
1. 타임라인 재구성: 사고 발생~탐지~봉쇄~복구 시각 정리
2. 근본 원인 확정
3. 대응 과정의 미흡 사항 식별
4. 재발 방지 조치 계획 수립 → SOP-CHANGE-001 또는 절차 개정 연계
5. 최종 사고 보고서(부록 A 완성) QA Manager 서명

## 6. 시크릿 침해 연계

시크릿 유출(비밀번호, API 키, TLS 개인 키 등)이 원인인 경우 **VAL-SEC-001 §6 시크릿 노출 시 대응** 절차를 병행 실행한다. 두 절차는 동시에 진행한다.
