## 1. 목적

본 SOP는 EDMS **운영 단계**에서 발생하는 일탈(절차 위반, 시스템 오류, 예기치 않은 동작)의 보고·분류·조사·시정 절차를 정의한다.

> **범위 구분**: ValidationPlan §일탈관리 절차는 검증(IQ/OQ/PQ) 단계에만 적용된다. 본 SOP는 시스템이 운영 릴리즈(Validation Report 승인) 이후 일상 운영 중 발생하는 일탈에 적용된다. 보안 사고(계정 침해, 데이터 위변조 등)는 SOP-INCIDENT-001 적용.

---

## 2. 범위

EDMS 운영 단계(Production 환경)에서 발생하는 다음 유형의 일탈에 적용된다:
- 절차 위반: SOP 미준수 행위 (예: 계정 공유, 비인가 접근 시도)
- 시스템 오류: 예상치 못한 시스템 동작 (예: PDF 변환 실패, 서명 오류)
- 데이터 무결성 우려: 데이터 불일치, 누락, 오류 발견
- 성능 저하: SLA 위반 (예: API 응답 2초 이상 지속, 시스템 다운 4시간 이상)

---

## 3. 용어 정의

| 용어 | 정의 |
|---|---|
| 일탈 (Deviation) | 규정된 절차·요구사항·시스템 동작으로부터의 이탈 |
| 시정조치 (Corrective Action) | 일탈 원인 제거를 위한 즉각적 또는 단기 조치 |
| 예방조치 (Preventive Action) | 재발 방지를 위한 시스템·절차 개선 조치 |
| 근본 원인 (Root Cause) | 일탈을 발생시킨 근본적 원인 (5-Why 방법론) |
| 효과성 검증 | 시정·예방조치가 재발을 방지했는지 확인 |

---

## 4. 일탈 분류 기준

| 심각도 | 기준 | 보고 기한 |
|---|---|---|
| **Critical** | GxP 데이터 손실 또는 위변조; Part 11 직접 위반; 전자서명 무효화; 감사추적 손상 | **24시간 이내** 보고 |
| **Major** | 절차 위반 (반복 또는 고의적); 시스템 다운 4시간 이상; 권한 오류 (비인가 접근 1건); 백업 실패 | **72시간 이내** 보고 |
| **Minor** | 사용자 실수 (자가복구 가능); 단순 기능 오류 (GxP 영향 없음); 일시적 성능 저하 | **7영업일 이내** 보고 |

> Critical 일탈은 자동으로 SOP-INCIDENT-001 보안사고와 경계를 확인한다:
> - 보안 침해(계정 탈취, 데이터 위변조)가 원인 → SOP-INCIDENT-001 발동
> - 시스템 결함 또는 절차 위반이 원인 → 본 SOP 계속 적용

---

## 5. 절차

### 5.1 일탈 발견 및 즉시 조치

1. 일탈을 발견한 사람은 즉시 업무를 중단하고 일탈 내용을 기록한다.
2. 즉각적인 영향 완화 조치를 취한다 (가능한 경우):
   - 시스템 오류: 오류 재현 방지 (예: 문제 버전 사용 중단)
   - 데이터 오류: 영향 받은 기록 격리 (상태를 DRAFT 이전 상태로 유지)
   - 절차 위반: 위반자에게 즉시 시정 요청
3. 부서 책임자 및 시스템 관리자에게 구두 또는 이메일로 즉시 통보한다.
4. 보고 양식(부록 A)을 작성하여 제출한다.

### 5.2 보고 및 등록

1. 일탈 발견자(또는 부서 책임자)가 **부록 A — 일탈 보고서**를 작성하여 QA에 제출한다.
2. QA가 일탈을 등록하고 심각도를 분류한다.
3. 조사 담당자를 지정한다:
   - Critical: QA + 시스템 관리자 공동
   - Major: 시스템 관리자 (QA 감독)
   - Minor: 담당 부서 또는 시스템 관리자

### 5.3 조사 — 근본 원인 분석 (Root Cause Analysis)

**5-Why 방법론** 적용:

```
예시:
일탈: 사용자 A가 자신의 권한 범위 밖의 문서에 서명함
Why 1: 왜 서명할 수 있었나? → 권한 매트릭스에 잘못된 권한이 설정되어 있었음
Why 2: 왜 잘못 설정됐나? → 부서 이동 시 이전 부서 권한을 제거하지 않았음
Why 3: 왜 제거하지 않았나? → SOP-USER-001 §5.2.2에 절차가 있으나 관리자가 미이행
Why 4: 왜 미이행했나? → 체크리스트 없이 구두로만 처리함
Why 5: (근본 원인) → 부서 이동 처리 절차에 체크리스트가 없었음
```

조사 결과를 부록 A 양식에 기록한다.

### 5.4 영향 범위 평가

- GxP 영향 여부: 해당 일탈이 GxP 데이터·문서의 신뢰성에 영향을 미치는가?
- 기록 유효성: 일탈 기간 동안 생성된 서명·기록이 여전히 유효한가?
- Critical 기능 영향: Part 11 §11.10 준수에 영향이 있는가?
- 규제 보고 의무: 식약처 등 외부 기관에 보고 의무가 있는가?

### 5.5 시정조치 (Corrective Action)

1. 단기 시정조치: 즉각 위험 제거 (예: 잘못된 권한 즉시 수정, 오류 버전 패치)
2. 장기 예방조치: 재발 방지 (예: 절차 개선, 시스템 통제 강화, 교육)
3. 시스템 코드 변경이 필요한 경우: SOP-CHANGE-001 변경 통제 적용
4. 조치 기한 및 담당자 지정

### 5.6 효과성 검증 (Effectiveness Check)

시정·예방조치 완료 후 30~60일 이내:
1. 동일 또는 유사 일탈이 재발했는지 확인
2. 적용된 통제(절차 변경, 시스템 패치 등)가 의도대로 작동하는지 검증
3. 결과를 부록 A 양식 하단에 기록

### 5.7 종결 (Closure)

모든 시정조치 완료 및 효과성 검증 후:
1. QA가 일탈 보고서를 검토하고 종결 서명한다.
2. 종결 기록을 EDMS에 저장한다 (카테고리: QA / 분류: 일탈기록).
3. 일탈 대장(부록 B)을 업데이트한다.

## 7. SOP-INCIDENT-001과의 경계

| 구분 | 처리 SOP |
|---|---|
| 보안 침해, 계정 탈취, 데이터 위변조 의심 | SOP-INCIDENT-001 (보안사고) |
| 시스템 결함, 절차 위반, 사용자 오류 | SOP-DEVIATION-001 (운영 일탈) |
| Critical 일탈에서 보안 침해가 원인으로 밝혀진 경우 | SOP-INCIDENT-001으로 에스컬레이션 |
