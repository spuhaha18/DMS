## 1. 목적

본 SOP는 EDMS에서 관리되는 **통제문서의 정기검토**와 **EDMS 시스템 자체에 대한 정기 재검토** 두 가지 활동을 정의한다.

통제문서 정기검토는 문서가 현행 절차·규정을 반영하고 있는지 주기적으로 확인한다. 시스템 정기 재검토는 EDMS가 검증 상태를 유지하고 있는지 연 1회 확인한다.

---

## 2. 범위

| 정기검토 유형 | 대상 | 주기 |
|---|---|---|
| **통제문서 정기검토** | EDMS에서 시행중(EFFECTIVE) 상태인 모든 통제문서 | 문서 카테고리별 설정 주기 (기본: 24개월) |
| **시스템 정기 재검토** | EDMS 시스템 자체 (소프트웨어, 설정, 검증 상태) | 연 1회 |

---

## 3. 용어 정의

| 용어 | 정의 |
|---|---|
| 정기검토 (Periodic Review) | 시행중 문서를 주기적으로 검토하여 현행성·적정성을 확인하는 활동 |
| 검토 주기 | 문서 카테고리별로 Admin이 설정한 정기검토 간격 (월 단위) |
| 검토 기산일 | 문서의 `effective_date` (시행일) — 이 날로부터 검토 주기를 계산 |
| 검토 만기일 | 검토 기산일 + 검토 주기 |
| 개정 (Revision) | 문서 내용의 실질적 변경이 필요한 경우 — 새로운 Draft 버전 생성 |
| 재확인 (Reconfirmation) | 내용 변경 없이 현행 유효함을 확인하고 문서 검토 주기를 갱신 |
| 폐기 (Retirement) | 더 이상 필요 없는 문서를 시스템에서 RETIRED 상태로 전환 |
| 시스템 재검토 | EDMS 소프트웨어·인프라·검증 상태에 대한 연 1회 재평가 |

## 4. 책임

### 4.1 통제문서 정기검토

| 역할 | 책임 |
|---|---|
| EDMS 시스템 | 검토 만기일 90/30/7/0일 전 담당자에게 자동 알림 발송 |
| 문서 소유자 (Owner) | 검토 만기일 내 검토 실시, 개정/재확인/폐기 결정 |
| Reviewer | 문서 소유자의 검토 의견 검토 서명 |
| QA | 전사 정기검토 현황 모니터링, 초과 문서 부서장 보고 |
| 부서 책임자 | 검토 지연 문서 독려, 폐기 결정 최종 승인 |

### 4.2 시스템 정기 재검토

| 역할 | 책임 |
|---|---|
| QA | 시스템 재검토 수행, 재검토 보고서 작성 |
| 개발 책임자 | 기술적 변경사항 검토 참여 |
| QA Manager | 재검토 보고서 최종 승인 |

## 5. 통제문서 정기검토 절차

### 5.1 정기검토 주기 설정

시스템 관리자(Admin)가 문서 카테고리별로 정기검토 주기를 설정한다.

| 카테고리 | 기본 검토 주기 | 비고 |
|---|---|---|
| SOP | 24개월 | 규정 변경 시 즉시 검토 트리거 가능 |
| Method | 24개월 | 분석 방법 개정 시 즉시 검토 |
| Specification | 24개월 | |
| Form | 24개월 | |

Admin 화면 → [카테고리 관리] → `review_period_months` 값 설정.

---

### 5.2 EDMS 자동 알림

EDMS는 `PeriodicReviewScheduler`가 매일 00:10 KST에 실행되어 아래 알림을 자동 발송한다.

| D-day | 알림 타입 | 수신자 |
|---|---|---|
| 검토 만기 90일 전 | 사전 안내 알림 | 문서 소유자 |
| 검토 만기 30일 전 | 검토 시작 촉구 알림 | 문서 소유자 + 부서 책임자 |
| 검토 만기 7일 전 | 긴급 알림 | 문서 소유자 + 부서 책임자 + QA |
| 검토 만기 당일 (D-0) | 기한 초과 알림 (매일 반복, 최대 30일) | 문서 소유자 + 부서 책임자 + QA |

---

### 5.3 검토 실시 (문서 소유자)

1. EDMS에서 정기검토 알림 확인.
2. 해당 문서의 최신 시행중 버전을 열람한다.
3. 아래 관점에서 문서 내용을 검토한다.

| 검토 관점 | 확인 내용 |
|---|---|
| 현행 절차 반영 | 실제 운영 절차와 문서 내용이 일치하는가 |
| 규정 변경 반영 | 관련 법령·규정 개정 사항이 반영되었는가 |
| 장비·시약 변경 | 문서에 기재된 장비·시약이 현재와 동일한가 |
| 조직 변경 반영 | 부서명, 직위, 담당자 정보가 현행인가 |
| 오탈자·오류 | 내용상 오류 또는 오탈자가 있는가 |

4. 검토 결과에 따라 아래 세 가지 중 하나를 결정한다.

---

### 5.4 검토 결정 유형 및 처리

#### 유형 A: 재확인 (내용 변경 불필요)

문서 내용이 현행 절차를 올바르게 반영하고 있어 변경이 불필요한 경우.

1. EDMS에서 해당 문서 → [정기검토 재확인] 버튼 클릭.
2. 검토 의견 입력: "내용 검토 완료, 변경 불필요"
3. 서명 다이얼로그에서 서명 (의미: REVIEWED).
4. 필요 시 Reviewer 추가 서명 (카테고리별 워크플로 설정에 따름).
5. 재확인 완료 → `effective_date`를 오늘 날짜로 갱신, 다음 검토 만기일 자동 재계산.

#### 유형 B: 개정 (내용 변경 필요)

문서 내용에 실질적 변경이 필요한 경우.

1. EDMS에서 해당 문서 → [개정 시작] 버튼 클릭 (T-06 전이).
2. 새로운 Draft 버전이 생성된다 (기존 시행중 버전 유지).
3. Author가 새 Draft를 수정하고 검토·승인 워크플로를 진행한다.
4. 새 버전이 시행중으로 전환되면 기존 버전은 자동으로 대체됨(SUPERSEDED) 처리된다.
5. 개정 완료 후 새 시행중 버전의 `effective_date`가 기산일이 되어 검토 주기가 재계산된다.

#### 유형 C: 폐기 (문서 사용 중단)

문서가 더 이상 필요 없는 경우 (해당 절차 폐지, 다른 SOP로 통합 등).

1. 문서 소유자가 폐기 필요성을 부서 책임자에게 보고하고 승인을 받는다.
2. EDMS에서 해당 문서 → [폐기 요청] 버튼 클릭.
3. 폐기 사유 입력 후 QA에게 폐기 서명 요청.
4. QA가 서명 (의미: RETIRED) → 문서 상태 = 폐기됨(RETIRED).
5. 폐기된 문서는 아카이브에 영구 보존되며, 시행중 문서 목록에서 제외된다.

---

### 5.5 정기검토 초과 문서 관리

**검토 만기 초과 기준**: D-0 알림 발송 이후 30일을 초과하여 처리되지 않은 문서.

| 기간 | 조치 |
|---|---|
| 만기 후 1~30일 | EDMS가 매일 알림 발송 (자동) |
| 만기 후 31~60일 | QA가 부서장에게 직접 통보, 조치 기한 부여 |
| 만기 후 61일+ | QA Manager 보고, 문서 사용 제한 여부 검토 (경우에 따라 해당 SOP 사용 금지 조치 가능) |

> **참고**: GxP 감사에서 "검토 기한 초과 SOP로 수행된 실험"은 데이터 신뢰성 문제로 지적될 수 있다.

## 6. EDMS 시스템 정기 재검토 절차

### 6.1 재검토 스케줄

**주기**: 연 1회 (시스템 릴리즈 기준일로부터 1년)  
**수행자**: QA (개발 책임자 참여)  
**산출물**: 시스템 재검토 보고서

---

### 6.2 재검토 항목

QA는 아래 항목을 검토하고 **시스템 재검토 보고서(부록 B)** 에 결과를 기록한다.

#### 6.2.1 변경 이력 검토

지난 1년간 Change Control(SOP-CHANGE-001)로 처리된 변경 이력을 검토한다.

| 확인 항목 | 내용 |
|---|---|
| Major 변경 건수 및 재검증 완료 여부 | 모든 Major CR에 대해 재검증이 수행되었는가 |
| Minor 변경 건수 | 적정 분류였는가 (사후 재평가) |
| 미종결 Deviation | 종결되지 않은 이탈 사항이 있는가 |

#### 6.2.2 감사로그 무결성 샘플 검증

```sql
-- 임의 날짜의 audit_logs 해시체인 연속성 샘플 검증
SELECT id, prev_hash, this_hash
FROM audit_logs
WHERE server_ts::date = '[샘플 날짜]'
ORDER BY id
LIMIT 20;
```

각 레코드의 `prev_hash`가 이전 레코드의 `this_hash`와 일치하는지 5건 이상 샘플 확인.

해시체인 검증 API 활용:
```
POST /api/v1/audit-logs/checkpoints/verify
body: { "from": "YYYY-MM-DD", "to": "YYYY-MM-DD" }
```

#### 6.2.3 WORM 앵커 파일 존재 확인

```bash
# 지난 30일 앵커 파일 존재 여부 확인
mc ls edms-audit-anchors/anchors/$(date +%Y)/$(date +%m)/
```

지난 30일 모든 날짜의 앵커 파일이 존재하는지 확인.

#### 6.2.4 백업 복구 훈련 수행 여부

SOP-BACKUP-001에 따른 반기 복구 훈련이 지난 1년간 2회 수행되었는지 확인.

#### 6.2.5 계정 접근 현황 (연간 종합)

SOP-USER-001의 분기 검토 4회 수행 여부 및 주요 이슈 확인.

#### 6.2.6 신규 규정 반영 필요 여부

지난 1년간 발표된 관련 규정 (FDA 가이드라인, EMA 가이드라인, GAMP 업데이트 등) 중 시스템에 영향을 미치는 사항이 있는지 검토.

#### 6.2.7 현행 URS 요건 충족 여부

시스템이 URS(VAL-URS-001)에 정의된 요건을 여전히 충족하고 있는지 확인. 요건 변경이 필요한 경우 URS 개정 및 Change Control 연계.

---

### 6.3 재검토 결론

| 결론 | 기준 | 후속 조치 |
|---|---|---|
| 검증 상태 유지 | 주요 이상 없음 | 다음 정기 재검토 일정 설정 |
| 부분 재검증 필요 | 미처리 변경 또는 규정 반영 필요 | Change Control + 영향 범위 재검증 |
| 전체 재검증 검토 | 시스템 대규모 변경 또는 중대 이탈 | QA Manager 및 경영진 보고 |
