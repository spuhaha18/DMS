## 1. 목적

본 SOP는 EDMS(Electronic Document Management System)에 대한 모든 변경을 체계적으로 관리하고, 검증 상태(Validated State)를 유지하기 위한 절차를 정의한다.

시스템 변경이 데이터 무결성, 전자서명 유효성, 감사추적 완결성, GxP 규정 준수에 영향을 미치지 않도록 통제한다.

---

## 2. 범위

본 SOP는 아래 EDMS 구성 요소에 대한 모든 변경에 적용된다.

| 대상 | 예시 |
|---|---|
| 소프트웨어 | 백엔드(Spring Boot), 프론트엔드(Vue 3), 라이브러리 업데이트 |
| 데이터베이스 | 스키마 변경(Flyway), 인덱스 추가, DB 설정 변경 |
| 인프라 | 서버 교체, OS 업그레이드, Nginx 설정 변경 |
| 미들웨어 | PostgreSQL 버전 업그레이드, MinIO 업그레이드, LibreOffice 업데이트 |
| 설정 | application.yml, 채번 템플릿, 워크플로 템플릿 |
| 본 SOP 자체 | SOP 내용 변경 |

**제외**: 초기 신규 도입(Validation Report로 관리), 장애 긴급 복구(SOP-DR-001 적용 후 사후 Change Control 수행)

---

## 3. 용어 정의

| 용어 | 정의 |
|---|---|
| 변경 요청 (CR, Change Request) | 시스템에 대한 수정을 공식적으로 요청하는 행위 |
| 영향 평가 (Impact Assessment) | 변경이 검증 상태, GxP 요건, 데이터 무결성에 미치는 영향을 평가하는 활동 |
| 주요 변경 (Major Change) | 검증 문서 갱신 및 재검증(OQ/PQ 일부)이 필요한 변경 |
| 경미한 변경 (Minor Change) | 영향 평가 후 OQ 일부 재수행 또는 재검증 없이 처리 가능한 변경 |
| 긴급 변경 (Emergency Change) | 즉각 조치가 필요한 보안 취약점·치명적 버그 수정 |
| 검증 상태 | 시스템이 URS 요건을 충족하고 GxP 규정을 준수함이 문서로 입증된 상태 |
| 재검증 | 변경 이후 영향 범위에 대해 IQ/OQ/PQ 일부를 재수행하는 활동 |

## 4. 책임

| 역할 | 책임 |
|---|---|
| 변경 요청자 | CR 양식 작성 및 제출, 기술적 배경 설명 |
| 개발 책임자 | 기술적 영향 평가, 구현, 코드 리뷰 |
| QA | 영향 평가 검토, 변경 분류, 재검증 범위 결정, CR 최종 기록 관리 |
| QA Manager | Major 변경 최종 승인, 긴급 변경 사후 승인 |
| IT/인프라 | 인프라 관련 변경 구현 |
| 시스템 관리자(Admin) | 설정 변경 구현 |

---

## 5. 절차

### 5.1 변경 요청 등록

1. 변경 요청자가 **CR 양식(부록 A)** 을 작성한다.
   - 변경 내용 상세 기술
   - 변경 이유 및 근거
   - 요청 우선순위 (Normal / Urgent)
2. CR 양식을 QA에게 제출한다 (이메일 또는 EDMS 내 문서로 제출).
3. QA가 CR 번호를 부여한다: `CR-YYYY-NNN` (예: CR-2026-001)
4. QA가 **CR 대장(부록 B)** 에 등록한다.

---

### 5.2 변경 분류 및 영향 평가

QA와 개발 책임자가 공동으로 아래 기준에 따라 변경을 분류하고 영향 평가를 수행한다.

#### 변경 분류 기준

| 분류 | 기준 | 처리 경로 |
|---|---|---|
| **Major** | - Critical 기능 변경 (전자서명, 감사추적, 상태머신, 권한 체크)<br>- DB 스키마 변경 (컬럼 추가·삭제·변경)<br>- 인프라 서버 교체, OS/PostgreSQL 메이저 버전 업그레이드<br>- URS 요건에 직접 영향<br>- MinIO Object Lock 정책 변경 | 5.3 (Major 승인 경로) |
| **Minor** | - 비GxP 기능 수정 (UI 레이아웃, 알림 텍스트 등)<br>- 버그 수정 (로직 변경 없는 수준)<br>- 성능 튜닝 (쿼리 최적화, 인덱스 추가)<br>- 라이브러리 패치 버전 업데이트 (x.y.Z)<br>- 워크플로 템플릿 설정 변경 | 5.4 (Minor 승인 경로) |
| **Emergency** | - 보안 취약점 즉시 패치<br>- 시스템 운영 불가 치명적 버그<br>- 데이터 무결성 위협 긴급 조치 | 5.5 (긴급 변경 경로) |

#### 영향 평가 항목

QA는 아래 항목을 검토하여 영향 평가 결과를 기록한다.

| 평가 항목 | 영향 있음 | 영향 없음 |
|---|---|---|
| 전자서명 기능 | ☐ | ☐ |
| 감사추적 완결성 | ☐ | ☐ |
| 문서 라이프사이클 상태머신 | ☐ | ☐ |
| 접근 권한 체크 | ☐ | ☐ |
| 데이터 무결성 (해시체인) | ☐ | ☐ |
| URS 요건 (해당 요건 번호: ) | ☐ | ☐ |
| 기존 검증 문서 갱신 필요 | ☐ | ☐ |
| 재검증 필요 | ☐ | ☐ |

---

### 5.3 Major 변경 승인 경로

```
CR 등록 (변경 요청자)
  ↓
영향 평가 완료 (QA + 개발 책임자, 5영업일 이내)
  ↓
재검증 계획 수립 (QA): 영향받는 OQ/PQ 케이스, 갱신할 검증 문서
  ↓
QA Manager 승인 (CR 양식 서명)
  ↓
구현 (개발팀 / IT)
  ↓
코드 리뷰 (개발 책임자)
  ↓
단위·통합 테스트 수행 (개발팀)
  ↓
재검증 수행 (QA): 영향 범위 OQ/PQ 케이스 재수행
  ↓
검증 문서 버전 갱신 (QA): 해당 FS/DS/RA/TM 개정
  ↓
운영 환경 배포 (IT)
  ↓
CR 종결 (QA): CR 대장 완료 처리
```

**Major 변경 목표 처리 기간**: CR 등록 후 30영업일 이내 (긴급도에 따라 조정)

---

### 5.4 Minor 변경 승인 경로

```
CR 등록 (변경 요청자)
  ↓
영향 평가 완료 (QA, 3영업일 이내)
  ↓
QA 승인 (CR 양식 서명)
  ↓
구현 및 단위테스트 (개발팀)
  ↓
QA 확인 테스트 (필요 시 — QA 판단)
  ↓
운영 환경 배포 (IT)
  ↓
CR 종결 (QA)
```

**Minor 변경 목표 처리 기간**: CR 등록 후 10영업일 이내

---

### 5.5 긴급 변경 경로

```
긴급 상황 감지 (개발팀 / IT / 보안팀)
  ↓
QA Manager에게 즉시 구두 보고 (1시간 이내)
  ↓
QA Manager 구두 승인
  ↓
즉시 구현 및 배포 (개발팀 / IT)
  ↓
사후 CR 양식 작성 (24시간 이내)
  ↓
사후 영향 평가 및 재검증 (QA, 5영업일 이내)
  ↓
QA Manager 사후 승인 서명
  ↓
CR 종결
```

> ⚠️ 긴급 변경은 반드시 사후 영향 평가를 수행한다. 영향 평가 없이 CR을 종결할 수 없다.

---

### 5.6 배포 절차

운영 환경 배포 시 아래 절차를 따른다.

1. **배포 전**: 현재 시스템 상태 스냅샷 기록 (SW 버전, DB 스키마 버전)
2. **배포 전**: DB 백업 수행 (SOP-BACKUP-001 §4.1)
3. **배포**: 사전 승인된 배포 스크립트 실행
4. **배포 후**: 핵심 기능 동작 확인 (헬스체크, 로그인, 감사로그 기록 확인)
5. **이상 발생 시**: 즉시 롤백 (사전 준비된 롤백 스크립트 실행)
6. **배포 기록**: CR 양식에 실제 배포일시, 배포자, 결과 기록
