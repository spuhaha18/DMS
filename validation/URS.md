## 1. 목적 및 범위

### 1.1 목적

본 문서는 제약회사 연구소(R&D)에서 사용할 EDMS(Electronic Document Management System)에 대한 사용자 요구사항을 정의한다. EDMS는 통제문서의 작성·검토·승인·시행·개정·폐기 전 라이프사이클을 디지털로 관리하며, 전자서명·감사추적·권한 기반 접근통제를 포함한다.

본 URS는 이후 수행될 기능명세(FS), 설계명세(DS), IQ/OQ/PQ 프로토콜 작성의 기준 문서이며, 추적성 매트릭스(Traceability Matrix)의 최상위 입력이 된다.

### 1.2 범위

**포함(in-scope)**:
- 통제문서 관리: SOP, Method, Specification, Form (Phase 1.a)
- 문서 라이프사이클 관리 (7단계)
- 전자서명 (21 CFR Part 11 §11.200 준수)
- 감사추적 (21 CFR Part 11 §11.10(e) 준수)
- 역할 기반 접근 통제 (RBAC 3차원)
- PDF 렌디션 생성 및 통제된 열람
- 사용자·역할·권한 관리
- 정기검토 알림
- 활성 통제문서 마이그레이션 도구

**포함하지 않음(out-of-scope, Phase 2+)**:
- Study Protocol, Study Report, IQ/OQ/PQ 장비검증 문서 (Phase 1.b/1.c)
- CMC 제출문서 (Phase 1.c)
- AD/LDAP SSO 연동 (Phase 1.c)
- 외부 LMS 연동 (Phase 2)
- Change Control / CAPA 워크플로 (Phase 2)
- LIMS / ELN / ERP 연동 (Phase 2)
- PKI 기반 디지털서명 (Phase 2)
- 모바일 앱 (Phase 2+)

### 1.3 대상 시스템 환경

- 배포 형태: On-Premise 폐쇄망 (사내 데이터센터)
- 사용자 규모: 200 ~ 500명
- 문서 규모: 초기 활성문서 ~250건, 운영 후 연 500건+ 증가 예상
- 클라이언트: 사내망에 연결된 PC 웹브라우저 (Chrome / Edge)

## 4. 시스템 개요

### 4.1 비즈니스 요구

제약회사 연구소는 GLP(비임상시험관리기준) 및 GMP(우수의약품 제조관리기준) 환경에서 운영된다. 이에 따라 통제문서(SOP, Method, Specification 등)의 작성·승인·발행·폐기 전 과정이 추적 가능하고 검증된 방식으로 관리되어야 한다.

현재 종이 또는 비구조적 파일공유(NAS) 방식의 문서 관리는 다음 문제를 야기한다:
- 현행 문서 버전을 알 수 없거나, 폐기된 문서가 여전히 사용되는 위험
- 서명 이력·검토 이력 등 감사증적 부재
- 규제기관(KFDA, FDA) 감사 시 문서 추적·제출에 과도한 시간 소요
- 무단 접근에 의한 신약 IP 유출 위험

### 4.2 시스템 목적

EDMS는 다음을 달성한다:
1. 모든 통제문서를 단일 시스템에서 라이프사이클 관리
2. 21 CFR Part 11 / EU Annex 11 준수 전자서명 및 감사추적 제공
3. RBAC 기반 접근통제로 IP 보호 및 협업 활성화
4. 감사관 대응 시간을 단위 → 분 단위로 단축

### 4.3 사용자 범주

| 역할 | 주요 행위 |
|---|---|
| Author (작성자) | 문서 작성, 초안 업로드, 개정 시작 |
| Reviewer (검토자) | 문서 내용 검토, 검토 완료 서명 또는 반려 |
| Approver (승인자) | 문서 승인 서명 또는 반려 |
| QA (품질보증 담당자) | QA 최종 승인, 발행(시행중 전환), 문서 회수 |
| RA (규제업무 담당자) | 규제 제출용 문서 조회, 다운로드 |
| Reader (열람자) | 유효 문서 열람, 교육 이수 확인 |
| Admin (시스템 관리자) | 사용자·역할·권한·마스터데이터 관리 |
| Auditor (감사관) | 읽기 전용, 감사로그·모든 버전 접근, 기간 제한 계정 |

## 5. UR-AUTH: 인증 및 세션

> **규제 근거**: 21 CFR Part 11 §11.10(d), §11.200(a), §11.300 / EU Annex 11 clauses 12, 19

---

**UR-AUTH-001**
**제목**: 사용자 인증

시스템은 사용자가 고유 사용자 ID와 비밀번호를 이용하여 로그인할 수 있어야 한다.

| 항목 | 내용 |
|---|---|
| 우선순위 | Critical |
| 규제 참조 | 21 CFR §11.300(a)(b) |
| 수락기준 | 유효 ID/PW로 로그인 성공; 잘못된 자격증명으로 로그인 거부 확인 |

---

**UR-AUTH-002**
**제목**: 비밀번호 복잡도

시스템은 다음 비밀번호 정책을 강제해야 한다:
- 최소 8자 이상
- 대문자, 소문자, 숫자, 특수문자 중 3종 이상 포함
- 최근 5개 비밀번호 재사용 불가
- 최초 로그인 또는 관리자 초기화 후 반드시 변경 강제

| 항목 | 내용 |
|---|---|
| 우선순위 | Critical |
| 규제 참조 | 21 CFR §11.300(b) |
| 수락기준 | 정책 위반 시 설정 거부 및 오류메시지 표시; 최근 비밀번호 재사용 거부 확인 |

---

**UR-AUTH-003**
**제목**: 로그인 실패 제한

연속 로그인 실패 5회 시 해당 계정을 자동으로 잠금(lock) 처리해야 한다. 잠금은 관리자가 해제하거나 30분 후 자동 해제된다. 잠금 이벤트는 감사로그에 기록된다.

| 항목 | 내용 |
|---|---|
| 우선순위 | Critical |
| 규제 참조 | 21 CFR §11.300(d) |
| 수락기준 | 5회 실패 후 계정 잠금, 이후 로그인 시도 거부, 감사로그 기록 확인 |

---

**UR-AUTH-004**
**제목**: 세션 자동 만료

비활성 상태 15분 경과 시 세션을 자동 종료하고 재로그인을 요구해야 한다.

| 항목 | 내용 |
|---|---|
| 우선순위 | Critical |
| 규제 참조 | 21 CFR §11.300(c) |
| 수락기준 | 15분 비활성 후 세션 만료 및 로그인 화면 리다이렉트 확인 |

---

**UR-AUTH-005**
**제목**: 인증 계층 추상화 (Phase 2 대비)

시스템의 인증 모듈은 인터페이스(AuthenticationProvider)로 추상화되어, 향후 AD/LDAP 연동 시 애플리케이션 코드 변경 없이 IdP를 교체할 수 있어야 한다. 사용자 테이블은 `external_id`, `auth_provider` 컬럼을 포함해야 한다.

| 항목 | 내용 |
|---|---|
| 우선순위 | High |
| 규제 참조 | — |
| 수락기준 | 구현 리뷰 시 인터페이스 분리 확인; `external_id` 컬럼 존재 확인 |

---

**UR-AUTH-006**
**제목**: 동시 세션 제한

단일 사용자 계정으로 동시에 2개 이상의 활성 세션을 허용하지 않아야 한다. 새 로그인 시 기존 세션을 만료시킨다.

| 항목 | 내용 |
|---|---|
| 우선순위 | High |
| 규제 참조 | EU Annex 11 clause 12 |
| 수락기준 | 두 번째 로그인 시 첫 번째 세션 만료 또는 경고 후 사용자 선택 |

## 6. UR-USER: 사용자 및 역할 관리

> **규제 근거**: 21 CFR Part 11 §11.10(d), §11.100 / EU Annex 11 clause 12

---

**UR-USER-001**
**제목**: 사용자 계정 생성

Admin은 신규 사용자 계정을 생성할 수 있어야 한다. 생성 항목: 사용자 ID(고유), 이름, 이메일, 부서, 직위, 역할 목록.

| 항목 | 내용 |
|---|---|
| 우선순위 | Critical |
| 수락기준 | 계정 생성 성공; 중복 ID 생성 시 오류 반환 확인 |

---

**UR-USER-002**
**제목**: 사용자 계정 비활성화

Admin은 퇴직·이동 등의 사유로 사용자 계정을 비활성화(disable)할 수 있어야 한다. 비활성화된 계정은 로그인이 불가하며, 계정 및 해당 사용자가 참여한 모든 이력(서명, 감사로그)은 보존된다. 계정은 삭제하지 않는다.

| 항목 | 내용 |
|---|---|
| 우선순위 | Critical |
| 규제 참조 | 21 CFR §11.10(d) |
| 수락기준 | 비활성화 후 로그인 거부; 해당 사용자의 과거 서명 이력 조회 가능 확인 |

---

**UR-USER-003**
**제목**: 역할 정의 및 관리

시스템은 다음 역할을 사전 정의하고, Admin이 추가·수정할 수 있어야 한다:
Author / Reviewer / Approver / QA / RA / Reader / Admin / Auditor

| 항목 | 내용 |
|---|---|
| 우선순위 | Critical |
| 수락기준 | 각 역할에 할당된 기능 권한 목록 UI에서 확인 가능 |

---

**UR-USER-004**
**제목**: 사용자 역할 배정

Admin은 사용자에게 하나 이상의 역할을 배정하거나 회수할 수 있어야 한다. 역할 변경은 감사로그에 기록된다.

| 항목 | 내용 |
|---|---|
| 우선순위 | Critical |
| 수락기준 | 역할 변경 후 해당 사용자의 접근 권한 즉시 반영; 감사로그 기록 확인 |

---

**UR-USER-005**
**제목**: Auditor 계정 기간 제한

Auditor 역할 계정은 유효기간(시작일·종료일)을 지정할 수 있어야 하며, 종료일 경과 시 자동으로 로그인이 불가해야 한다.

| 항목 | 내용 |
|---|---|
| 우선순위 | High |
| 수락기준 | 유효기간 만료 후 로그인 거부 확인 |

## 7. UR-ACC: 접근 통제

> **규제 근거**: 21 CFR Part 11 §11.10(d) / EU Annex 11 clause 12

---

**UR-ACC-001**
**제목**: RBAC 3차원 권한 매트릭스

시스템의 접근 통제는 다음 3차원으로 구성된다:
- 차원 1: 기능 역할 (Functional Role)
- 차원 2: 문서 카테고리 (Document Category Scope)
- 차원 3: 조직 범위 (Organizational Scope — 부서 또는 프로젝트)

예: "Reviewer × SOP × 약리연구부" = 약리연구부의 SOP에 대해서만 검토 권한 부여.

| 항목 | 내용 |
|---|---|
| 우선순위 | Critical |
| 수락기준 | A부서 문서를 B부서 사용자가 접근 시 거부됨을 확인; 권한 매트릭스 UI에서 설정·조회 가능 |

---

**UR-ACC-002**
**제목**: 기밀 프로젝트 문서 격리

## 8. UR-DOC: 문서 관리

> **규제 근거**: 21 CFR Part 11 §11.10(b), §11.70 / EU Annex 11 clause 7, 8

---

**UR-DOC-001**
**제목**: 통제문서 카테고리

Phase 1.a에서 시스템은 다음 문서 카테고리를 지원해야 한다:
- SOP (Standard Operating Procedure, 표준작업서)
- Method (분석법)
- Specification (규격서)
- Form (양식/템플릿)

각 카테고리는 카테고리 코드(예: SOP, MET, SPC, FRM)를 가지며, Admin이 마스터데이터에서 추가·관리할 수 있어야 한다.

| 항목 | 내용 |
|---|---|
| 우선순위 | Critical |
| 수락기준 | 4개 카테고리 각각에 대해 문서 생성 가능; 카테고리 코드 UI에서 관리 가능 |

---

**UR-DOC-002**
**제목**: 카테고리별 자동 채번 (포맷 템플릿)

문서 등록 시 시스템은 해당 카테고리의 채번 포맷 템플릿에 따라 문서번호를 자동으로 부여해야 한다. 사용자가 직접 입력·수정할 수 없으며, 한 번 부여된 번호는 변경·재사용하지 않는다.

**채번 포맷 템플릿 구조 (Admin이 카테고리별로 설정):**

포맷 문자열은 다음 플레이스홀더와 고정 텍스트, 구분자(기본값 `-`)의 조합으로 정의한다.

| 플레이스홀더 | 의미 | 예시 |
|---|---|---|
| `{TYPE}` | 카테고리 코드 (예: SOP, MET, SP, FRM) | SOP |
| `{DEPT}` | 부서 코드 (예: QC, RD, TOX) | QC |
| `{PROD}` | 제품/과제 코드 (문서 작성 시 선택) | ABC123 |
| `{YEAR}` | 시스템 현재 연도 (4자리) | 2026 |
| `{SEQ:N}` | 일련번호 (N자리 제로패딩) | 001 |
| 고정 텍스트 | 포맷 문자열에 직접 입력 | F, SP, PROT 등 |

**카운터 범위(scope) 설정:**

각 카테고리는 일련번호를 어떤 단위로 독립 관리할지 설정한다.

| scope 값 | 설명 | 예시 |
|---|---|---|
| `PER_DEPT` | 부서×카테고리별 독립 순번 | SOP-QC-001, SOP-RD-001 각자 1번부터 |
| `PER_PRODUCT` | 제품코드×카테고리별 독립 순번 | MET-ABC123-001 |
| `PER_YEAR` | 연도×카테고리별, 연초 자동 초기화 | SP-2026-001 |
| `GLOBAL` | 전사 단일 순번 | FRM-001 |

**카테고리별 설정 예시:**

```
SOP   : format="{TYPE}-{DEPT}-{SEQ:3}",  scope=PER_DEPT
Method: format="{TYPE}-{PROD}-{SEQ:3}",  scope=PER_PRODUCT
Form  : format="{DEPT}-F-{SEQ:3}",       scope=PER_DEPT
Spec  : format="SP-{YEAR}-{SEQ:3}",      scope=PER_YEAR
```

→ SOP-QC-001 / MET-ABC123-001 / QC-F-001 / SP-2026-001

포맷 템플릿 변경 이력은 감사로그에 기록되며, 변경 이전에 이미 부여된 번호에는 소급 적용되지 않는다.

| 항목 | 내용 |
|---|---|
| 우선순위 | Critical |
| 규제 참조 | 21 CFR §11.10(a) — 정확하고 완전한 기록 |
| 수락기준 | SOP·Method·Form·Spec 각 카테고리에서 서로 다른 포맷으로 번호 자동 부여 확인; 동일 scope 단위에서 중복 번호 발생 없음 확인; 포맷 변경 감사로그 기록 확인 |

## 9. UR-LCY: 문서 라이프사이클

> **규제 근거**: 21 CFR Part 11 §11.10(e), §11.50 / EU Annex 11 clause 10

---

**UR-LCY-001**
**제목**: 7단계 라이프사이클 상태 정의

시스템은 다음 7개의 상태를 지원해야 한다:

| 상태 | 설명 |
|---|---|
| 초안 | 작성 중인 문서. Author만 편집 가능. 열람 제한. |
| 검토중 | 지정된 검토자에게 검토 요청된 상태. 편집 불가. |
| 승인중 | 지정된 승인자에게 승인 요청된 상태. 편집 불가. |
| 시행중 | 최종 승인되어 유효한 상태. 열람 가능. 카테고리×부서당 1개만 존재. |
| 개정중 | 기존 시행중 버전을 유지하면서 개정 중인 상태. 신규 초안 병행 존재. |
| 대체됨 | 신 버전의 시행중 진입으로 대체된 상태. 열람은 가능(이력 조회)하나 검색 기본 제외. |
| 폐기됨 | 더 이상 유효하지 않고 사용이 중단된 상태. 영구 보존. |

| 항목 | 내용 |
|---|---|
| 우선순위 | Critical |
| 수락기준 | 각 7개 상태에서의 전이 가능·불가 케이스 모두 OQ로 검증 |

---

**UR-LCY-002**
**제목**: 상태 전이 매트릭스

시스템은 다음 상태 전이만 허용해야 한다. 정의되지 않은 전이는 거부한다. 워크플로 단계 수와 역할은 카테고리별 워크플로 템플릿(UR-LCY-007)에서 결정되며, 상태 전이의 트리거 조건은 해당 템플릿을 따른다.

| From → To | 허용 조건 |
|---|---|
| 초안 → 검토중 | Author 제출 + Author 서명; 문서에 워크플로 인스턴스 생성됨 |
| 검토중 → 초안 | 임의 검토 단계의 담당자가 반려 |
| 검토중 → 승인중 | 현재 검토 단계의 모든 지정 서명자 서명 완료 (병렬 설정 시 모두) |
| 승인중 → 검토중 | 임의 승인 단계의 담당자가 반려 |
| 승인중 → 시행중 | 워크플로 템플릿의 마지막 단계 서명자 전원 서명 완료 (QA 단계 있는 경우 QA 포함) |
| 시행중 → 개정중 | Approver 이상 권한 사용자가 개정 시작 (새 초안 생성) |
| 개정중 → 시행중 | 새 버전 시행중 전환 시 기존 시행중는 자동으로 대체됨 전환 |
| 시행중 → 폐기됨 | 워크플로 템플릿의 폐기 권한 역할 사용자가 폐기 서명 |
| 대체됨 → (없음) | 불변 (아카이브) |

| 항목 | 내용 |
|---|---|
| 우선순위 | Critical |
| 규제 참조 | 21 CFR §11.50 |
| 수락기준 | 각 전이 허용/거부 케이스를 OQ에서 개별 검증; 정의되지 않은 전이 시도 시 HTTP 422 반환 |

---

**UR-LCY-003**
**제목**: 시행중 단일성 보장

동일 문서번호에 시행중 상태인 버전은 **정확히 1개만** 존재해야 한다. 신 버전이 시행중으로 전환되는 트랜잭션은 원자적으로 처리되며, 이전 시행중 버전은 즉시 대체됨으로 변경된다.

| 항목 | 내용 |
|---|---|
| 우선순위 | Critical |
| 수락기준 | DB 레벨 제약(unique partial index 또는 트랜잭션)으로 동시에 2개 시행중 불가 확인 |

---

**UR-LCY-004**
**제목**: 개정 중 기존 시행중 열람 유지

개정중 상태에서도 기존 시행중 버전은 계속 열람 가능해야 한다. 진행 중인 연구·실험이 중단되지 않아야 한다.

| 항목 | 내용 |
|---|---|
| 우선순위 | Critical |
| 수락기준 | 개정중 상태에서 기존 시행중 버전 열람 성공 확인 |

---

**UR-LCY-005**
**제목**: 반려 사유 필수 입력

검토자 또는 승인자가 문서를 반려(reject)할 때 반려 사유를 필수로 입력해야 한다. 반려 사유는 감사로그 및 문서 이력에 기록된다.

| 항목 | 내용 |
|---|---|
| 우선순위 | Critical |

| 수락기준 | 사유 미입력 시 반려 처리 거부; 입력된 사유 감사로그에서 조회 가능 확인 |

---

**UR-LCY-006**
**제목**: 워크플로 인스턴스 생성 (담당자 지정)

Author가 문서를 제출할 때, 카테고리 워크플로 템플릿(UR-LCY-007)의 각 단계에 실제 담당자를 지정해야 한다. 지정 규칙:

- 각 단계에서 지정 가능한 역할은 템플릿이 정의한 범위 내로 제한된다.
- 템플릿에서 `auto_assign` 으로 설정된 단계(예: QA 단계)는 시스템이 해당 역할을 가진 사용자 풀에서 자동 지정하며 Author가 변경할 수 없다.
- 각 단계에 최소 1명 이상을 지정해야 하며, 템플릿에 설정된 최소 인원(min_signers)을 충족해야 한다.
- 병렬(parallel) 단계는 지정된 모든 서명자가 서명 완료해야 다음 단계로 전환된다.

| 항목 | 내용 |
|---|---|
| 우선순위 | Critical |
| 수락기준 | 담당자 미지정 단계가 있을 경우 제출 거부 확인; auto_assign 단계의 담당자 변경 시도 시 거부 확인; 병렬 단계에서 1명만 서명 후 전환 시도 시 거부 확인 |

---

**UR-LCY-007**
**제목**: 카테고리별 워크플로 템플릿 정의

Admin은 각 문서 카테고리에 대해 워크플로 템플릿을 정의할 수 있어야 한다. 템플릿은 다음 구조를 가진다:

```
워크플로 템플릿 (Admin 설정):
  카테고리: SOP
  단계 목록:
    Step 1 - type: REVIEW,    role: REVIEWER,  min_signers: 1, parallel: true,  auto_assign: false
    Step 2 - type: APPROVAL,  role: APPROVER,  min_signers: 1, parallel: false, auto_assign: false
    Step 3 - type: APPROVAL,  role: QA,        min_signers: 1, parallel: false, auto_assign: true,  qa_required: true
```

각 단계(step)의 설정 항목:
- `type`: REVIEW 또는 APPROVAL
- `role`: 해당 단계에 배정 가능한 역할 (REVIEWER / APPROVER / QA / RA 등)
- `min_signers`: 최소 서명자 수
- `parallel`: true면 지정된 모든 서명자가 동시에 서명 가능 (병렬), false면 순차
- `auto_assign`: true면 시스템이 해당 역할 보유자를 자동 배정하고 Author가 변경 불가
- `qa_required`: true인 단계가 있어야 시스템이 시행중 전환을 허용

템플릿은 버전 관리되며, 템플릿 변경 이력은 감사로그에 기록된다. 이미 진행 중인 워크플로에는 변경된 템플릿이 소급 적용되지 않는다.

| 항목 | 내용 |
|---|---|
| 우선순위 | Critical |
| 규제 참조 | EU Annex 11 clause 10; GAMP 5 |
| 수락기준 | Admin이 SOP/Method/Spec/Form 각각 다른 템플릿 설정 가능 확인; 템플릿 변경 이력 감사로그 기록 확인; 진행 중 워크플로에 소급 미적용 확인 |

---

**UR-LCY-008**
**제목**: QA 단계 선택적 구성

카테고리 워크플로 템플릿에서 QA 서명 단계는 선택 사항이다. QA 단계가 없는 카테고리(예: 일부 Form)는 QA 서명 없이 최종 Approver 서명만으로 시행중 전환이 가능해야 한다. QA 단계 포함 여부는 템플릿의 `qa_required` 플래그로 결정된다.

단, 규제 제출 대상 문서(예: GLP Study Report, IND/NDA 관련 문서)는 반드시 QA 단계를 포함하는 템플릿을 사용해야 하며, 시스템은 이를 카테고리 수준에서 강제할 수 있어야 한다 (`qa_mandatory` 플래그).

| 항목 | 내용 |
|---|---|
| 우선순위 | Critical |
| 규제 참조 | 21 CFR §11.50; GLP 요건 |
| 수락기준 | QA 단계 없는 템플릿의 문서가 Approver 서명 후 시행중 전환 가능 확인; qa_mandatory 설정 카테고리에서 QA 단계 제거 시도 시 Admin에게 경고 표시 확인 |

---

**UR-LCY-009**

## 17. UR-SEC: 보안

> **규제 근거**: 21 CFR Part 11 §11.10(c)(d)(g) / EU Annex 11 clause 12

---

**UR-SEC-001**
**제목**: 전송 암호화

모든 클라이언트-서버 통신은 TLS 1.2 이상으로 암호화되어야 한다. 사내 CA가 발급한 인증서를 사용할 수 있다.

| 항목 | 내용 |
|---|---|
| 우선순위 | Critical |
| 수락기준 | HTTP(평문) 접속 시 HTTPS 리다이렉트 확인; TLS 버전 1.2+ 확인 |

---

**UR-SEC-002**
**제목**: 비밀번호 저장 암호화

사용자 비밀번호는 bcrypt 또는 Argon2id 등 단방향 해시 알고리즘으로 저장해야 한다. 평문 비밀번호를 DB에 저장하거나 로그에 기록하지 않는다.

| 항목 | 내용 |
|---|---|
| 우선순위 | Critical |
| 수락기준 | DB password 컬럼이 해시값임을 확인; 로그 파일에 평문 비밀번호 미포함 확인 |

---

**UR-SEC-003**
**제목**: 파일 직접 접근 차단

MinIO 스토리지의 파일은 애플리케이션 API를 통해서만 접근 가능해야 한다. 직접 MinIO URL로의 파일 접근은 차단되어야 한다.

| 항목 | 내용 |
|---|---|
| 우선순위 | Critical |
| 수락기준 | 직접 MinIO Presigned URL 없이 파일 URL 직접 접근 시 403 확인 |

---

**UR-SEC-004**
**제목**: 타임스탬프 서버 기반

모든 서명·감사로그의 타임스탬프는 서버 시각(NTP 동기화된)을 사용해야 한다. 클라이언트 제공 시각은 사용하지 않는다.

| 항목 | 내용 |

| 수락기준 | 클라이언트 시각 조작 후 서명 시 서버 시각이 기록됨을 확인 |

---

**UR-SEC-005**

모든 사용자 입력값은 서버 측에서 검증되어야 한다. SQL 인젝션, XSS, 경로 탐색 공격 등의 입력 패턴은 거부되어야 한다.
